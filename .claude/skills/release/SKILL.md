# Release Process

Releases use `cargo release` to bump version, commit, tag, and push. The tag push triggers a GitHub Actions workflow (`cargo-dist`) that builds cross-platform binaries and creates a GitHub Release.

## Prerequisites

Before releasing, verify:

1. **All tests pass:**
```bash
cargo test
cargo clippy -- -D warnings
cargo fmt --check
```

2. **Main branch is clean:**
```bash
git status  # must be clean
git log --oneline -5  # review recent commits
```

3. **CI is green** on the latest commit:
```bash
gh run list --limit 1
```

## Making a release

### Dry run first (always)

```bash
cargo release <level>  # patch, minor, major, or rc
```

Without `--execute`, this shows what would happen without doing anything. Review the output with the user before proceeding.

### Execute

Only after the user confirms the dry run:

```bash
cargo release <level> --execute
```

This will:
1. Bump the version in `Cargo.toml`
2. Create a commit: `chore: Release mdvs version <new_version>`
3. Create a git tag: `v<new_version>`
4. Push the commit and tag to `origin`

The tag push triggers `.github/workflows/release.yml`, which builds binaries for 4 platforms and creates a GitHub Release.

### Release levels

| Command | Example | Use case |
|---------|---------|----------|
| `cargo release patch --execute` | 0.1.0 → 0.1.1 | Bug fixes |
| `cargo release minor --execute` | 0.1.0 → 0.2.0 | New features |
| `cargo release major --execute` | 0.1.0 → 1.0.0 | Breaking changes |
| `cargo release rc --execute` | 0.1.0 → 0.1.1-rc.1 | Prerelease / testing |

### After pushing

Monitor the release build:

```bash
gh run list --limit 1                  # find the run ID
gh run view <run-id>                   # check job status
gh run watch <run-id>                  # live follow (optional)
```

If a build fails:
```bash
gh run view --job=<job-id> --log-failed  # see failure logs
```

After success, verify the GitHub Release:
```bash
gh release view v<version>
```

## Configuration

- **`release.toml`** — cargo-release config (`publish = false`, `push-remote = "origin"`)
- **`Cargo.toml`** — version field, `[profile.dist]`, `[package.metadata.dist]`, `[workspace.metadata.dist]`
- **`.github/workflows/release.yml`** — auto-generated by `dist generate`, do not hand-edit

## Important notes

- `publish = false` — no crates.io publishing. Releases are GitHub-only for now.
- Tag format is `v{version}` (e.g., `v0.1.0`). This is the default — no prefix config needed.
- Prerelease tags (e.g., `v0.1.0-rc.1`) create a GitHub Release marked as prerelease.
- If cargo-dist config changes, regenerate the workflow: `dist generate`
- Full spec: `docs/spec/release.md`
