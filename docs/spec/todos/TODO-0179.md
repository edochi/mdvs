# TODO-0179: Module split pass

## Problem

Per an outside review (2026-06-04, action item 3), the three
largest modules in `crates/mdvs/src/` are big enough to read as a
classic LLM tell:

| file | lines |
|---|---|
| `cmd/build.rs` | 2,283 |
| `schema/json_schema.rs` | 1,972 |
| `index/backend.rs` | 1,521 |

The accretion is a real thing each module did honestly — each wave
added one more concern to `build_core`, each Wave-B step added a
translator branch to `json_schema.rs`, each Lance API call landed
in `backend.rs`. Result: long files that a skeptical reader skims,
sees the line count, and dismisses.

## Approach

Split each of the three (and any other file over ~600 lines) into a
small sub-module tree that follows the natural seams already
present in the code. Mechanical refactor only — **no behavior
change**, **no test changes** (except path imports), and the public
API surface stays identical.

### `cmd/build.rs` (2,283 → target ~3–5 files under ~600 each)

Natural seams already visible in the file:
- The classify types + `classify_files` function → `cmd/build/classify.rs`.
- The embed types + `embed_file` function → `cmd/build/embed.rs`.
- The `write_index` dispatch + `write_index_incremental` glue (NB:
  the backend-side incremental code stays in `index/backend.rs`) →
  `cmd/build/write.rs`.
- `mutate_config`, `normalize_revision`, `detect_config_changes` →
  `cmd/build/config_mutate.rs`.
- The orchestrator `build_core` + `run` + `pub use`s stay in the
  small `cmd/build.rs` (or `cmd/build/mod.rs`).

### `schema/json_schema.rs` (1,972 → ~3 files)

Natural split:
- `dsl_to_canonical` + helpers → `schema/json_schema/to_canonical.rs`.
- `canonical_to_dsl` + helpers → `schema/json_schema/from_canonical.rs`.
- `validate_mdvs_schema` gate + the keyword allow/deny lists →
  `schema/json_schema/validate.rs`.
- `compute_schema_hash` + `extract_leaf_schemas` + the small shared
  helpers stay in the parent `mod.rs`.

### `index/backend.rs` (1,521 → ~2 files)

- `Backend` enum + `LanceBackend` struct + write paths
  (`write_index`, `write_index_incremental`, `build_indexes`) →
  stays in `backend.rs`.
- Search dispatch (`SearchMode` matching, `nearest_to` / FTS /
  hybrid + `--where` translation) → `index/backend/search.rs`.
- The `read_file_index` / `read_chunk_rows` / `read_metadata` reader
  methods → `index/backend/read.rs`.

### Other files over ~600 lines

Audit during the split — anything else over the threshold gets the
same treatment. Likely candidates: `discover/infer/mod.rs`,
`cmd/check.rs` post the post-TODO-0172 growth.

## Verification

- No file under `crates/mdvs/src/` exceeds ~600 lines (modulo
  monster generated tables that legitimately can't be split).
- `cargo test` — full suite green, no test code changed beyond
  imports.
- `cargo clippy --all-targets` clean.
- `cargo fmt`.
- `git diff` shows pure moves: `git diff --find-renames=50%` should
  show high rename ratios on each split.
- Public API surface unchanged: `cargo doc --no-deps` produces the
  same items (spot-check `mdvs::cmd::build::build_core`,
  `mdvs::index::backend::Backend`, `mdvs::schema::json_schema::*`).
- No behavior regression on `example_kb` (check / build / search
  unchanged).

## Out of scope

- Renaming items or changing API shape.
- Removing or merging functionality.
- Doc updates beyond what the move forces (mostly intra-doc
  `crate::cmd::build::...` paths).
- Trimming or rewriting the function bodies themselves — that's
  fair-game in a separate cleanup pass, not here. This TODO is
  about file shape, not code quality.

## Impact

Per the reviewer, "the single highest-impact `doesn't look LLM-
written` change." Multiple smaller files signal the kind of
human-rotated work a reader expects to see; long monoliths signal
the kind they don't.
