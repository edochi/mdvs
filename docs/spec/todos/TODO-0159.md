# TODO-0159: --where on an Array(Float) field panics inside lance-encoding

## Status

**Mitigated 2026-05-23.** The translator now rejects `--where` references to
`Array(Float)` (`List<Float*>`) fields with a clean, actionable error across
all modes (`f3f9b61` family — actually the commit landing the guard). The
underlying Lance bug remains; revisit on a `lancedb` bump. Priority lowered
from `medium` to `low` since the user-visible severity (crash/hang → clear
error) is fixed.

## Summary

A `--where` clause that references an `Array(Float)` (Arrow `List<Float64>`)
frontmatter field panics inside LanceDB's encoding layer. In semantic mode
the panic surfaces on `main` (process exits). In fulltext/hybrid mode it
panics on a `tokio-rt-worker`, leaving the main task awaiting a future that
never resolves — **the process hangs**. Array(String) fields are unaffected;
normal searches (no such filter) are fine because the column is not projected.

## Details

### Repro (example_kb, field `measurement_values: Array(Float)`)

```
mdvs search x example_kb --where "array_has(measurement_values, 0.5)"
mdvs search x example_kb --where "measurement_values IS NOT NULL"
mdvs search x example_kb --where "measurement_values IS NULL"
```

All three panic:

```
thread 'main' panicked at lance-encoding-6.0.0/src/encodings/logical/primitive.rs:2505:45:
called `Option::unwrap()` on a `None` value
```

### Scope (measured 2026-05-22)

- Triggers for **any** `--where` predicate that references a `List<Float64>`
  field (membership, null checks — anything that makes Lance scan/decode the
  column to evaluate the filter).
- `Array(String)` (`List<Utf8>`, e.g. `tags`, `measurement_timestamps`)
  filters work fine (`array_has(tags, 'x')` → correct hits).
- `build`, `check`, and normal `search` (no float-array filter) are
  unaffected — search projects only
  `[file_id, filepath, start_line, end_line, chunk_text]`, so the float-array
  column is only decoded when a `--where` references it.
- The `embedding` column is `FixedSizeList<Float32>` and is fine — this is
  specific to the variable-length `List<Float64>` decode path.

### Why it happens

A LanceDB / lance-encoding 6.0 bug decoding `List<Float64>` (nested in the
`data` Struct) during filter evaluation. Not fixable in mdvs directly.

### Options

1. **Upstream / version bump** — retest after a `lancedb` upgrade; this may be
   fixed in a newer lance-encoding. Lowest effort, just gated on a bump.
2. **Schema-aware guard (mitigation)** — the `--where` translator already
   knows the frontmatter field names (data Struct children); extend it to also
   know their types and return a clean error
   ("filtering on Array(Float) fields is not supported — LanceDB limitation")
   when a clause references an `Array(Float)` field, turning the panic into a
   graceful user error. Avoids the crash without fixing Lance.
3. **Storage change** — store float arrays differently (e.g. fixed-size or a
   different encoding). Heavy; not worth it for a niche filter.

Recommended: try (1) on the next lancedb bump; (2) is already in place.

## What's now in place (option 2 mitigation)

`index/backend.rs` learned `float_list_child_names(schema)`, which scans the
`data` Struct for children of type `List<Float*>`. `translate_where_to_struct`
takes that set as a new param and returns an `Err` early if any matched
identifier in the clause names one of those fields (handling both bare names
and pre-qualified `data.<name>` references). The user sees:

```
Error: filtering on Array(Float) field 'measurement_values' is not supported
in --where (a LanceDB decode limitation). Filter on a different field or
store the values in a parallel scalar field.
```

instead of a crash/hang. Covered by translator unit tests +
`integration_array_float_filter_errors_not_panics` across all three modes.

## Files likely touched (for the eventual Lance-side fix)

- (upstream) `lance-encoding` `FixedFullZipDecoder::slice_next_task` at
  `primitive.rs:2505` (drain of a `StructuralListDecoder` over a sparse
  `List<Float64>`). Stack: `slice_next_task → drain → drain` then up through
  the structural list decoder.
- (mdvs cleanup, once Lance is fixed) drop `float_list_child_names` + the
  early-error in `translate_where_to_struct`, and remove the integration test
  asserting the error.
