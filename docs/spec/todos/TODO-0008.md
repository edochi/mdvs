# TODO-0008: Support value boundary constraints on numeric fields

## Summary

Add `min` and `max` constraints for range validation during check. Builds on the constraint infrastructure from TODO-0006. Supports Integer, Float, Array(Integer), and Array(Float) fields. Bounds are inclusive. Optionally inferred via `update reinfer --range`.

## Details

### Supported types

- `Integer` ÔÇö scalar, both bounds must be integers
- `Float` ÔÇö scalar, bounds can be integer or float (widened to float for comparison)
- `Array(Integer)` ÔÇö element-level, same rules as Integer
- `Array(Float)` ÔÇö element-level, same rules as Float

For arrays, each element is checked against the bounds individually. If any element is out of range, the violation lists the offending elements (same pattern as `InvalidCategory` on arrays).

### Type matching and widening

Reuse the existing widening algorithm for comparisons:

- Integer field + Integer bound Ôćĺ compare as i64
- Float field + Float bound Ôćĺ compare as f64
- Float field + Integer bound Ôćĺ widen bound to f64, compare as f64

**Config error** if the field is Integer (or Array(Integer)) but a bound is Float. This catches likely mistakes ÔÇö a float bound on an integer field is almost certainly wrong (e.g., `min = 0.5` on an Integer field). The user should write `min = 1` instead.

### Constraint kind

New `ConstraintKind::Range` variant on the existing enum:

```rust
enum ConstraintKind {
    Categories(Vec<toml::Value>),
    Range { min: Option<toml::Value>, max: Option<toml::Value> },
}
```

**`validate_for_type()`**: returns error for non-numeric types (Boolean, String, Object, and arrays of non-numeric types). Also returns error if a Float bound is used on an Integer field.

**`validate_value()`**: checks numeric value falls within `[min, max]` (inclusive). One or both bounds may be present. Returns `ConstraintViolation` on failure.

**`conflicts_with()`**: returns error when paired with `Categories` on the same field (redundant ÔÇö if you enumerate all allowed values, a range is meaningless). Returns `None` when paired with `Length` (different domains ÔÇö range constrains value, length constrains size).

### Serde layer

New fields on `Constraints` struct:

```rust
struct Constraints {
    categories: Option<Vec<toml::Value>>,
    #[serde(default, skip_serializing_if = "Option::is_none")]
    min: Option<toml::Value>,
    #[serde(default, skip_serializing_if = "Option::is_none")]
    max: Option<toml::Value>,
}
```

`active()` produces `ConstraintKind::Range` when either `min` or `max` is `Some`.

### TOML representation

```toml
# Scalar Integer with both bounds
[[fields.field]]
name = "rating"
type = "Integer"

[fields.field.constraints]
min = 1
max = 5

# Scalar Float with lower bound only
[[fields.field]]
name = "temperature"
type = "Float"

[fields.field.constraints]
min = 273.15

# Float field with integer bound (widened to float)
[[fields.field]]
name = "score"
type = "Float"

[fields.field.constraints]
min = 0
max = 100

# Array of Integers with range
[[fields.field]]
name = "ratings"
type = { array = "Integer" }

[fields.field.constraints]
min = 1
max = 10
```

### Validation

New `ViolationKind::OutOfRange`:

- **rule**: `min = 1, max = 5`
- **detail** (scalar): `got 0`
- **detail** (array): `elements out of range: [0, -1]`

### Inference

Range constraints are **not** inferred automatically by default ÔÇö unlike categories, there's no good heuristic for deciding a field "should" have bounds.

However, bounds **can** be inferred on demand via `update reinfer` using the `--with` flag (see CLI design below).

### Conflicts

- `categories` + `range` on the same field Ôćĺ config error (redundant ÔÇö enumerated values already define the range)
- `range` + `length` on the same field Ôćĺ no conflict (different domains)

### Config load validation

`validate_config()` (already called on mdvs.toml load) catches:

1. Range on a non-numeric field type Ôćĺ error
2. Float bound on an Integer field Ôćĺ error
3. `min > max` when both are present Ôćĺ error
4. Range + categories on the same field Ôćĺ error

### Storage/Query

No change. Constraints are validation-only.

## Implementation

### Wave 2 ÔÇö Inference + CLI rework + docs

Inference of min/max from observed values, plus a CLI rework that replaces the per-kind boolean flags (`--categorical`, `--no-categorical`, `--range`, `--no-range`) with a single `--with` flag taking a comma-separated list of constraint kinds.

**Files (already done in initial wave 2 commit, partially):**

- `src/discover/infer/constraints/range.rs` ÔÇö new submodule with `infer()` (DONE)
- `src/discover/infer/constraints/mod.rs` ÔÇö `infer_range()` orchestrator (DONE)

**CLI rework ÔÇö replace boolean flags with `--with`:**

The current design (`--categorical` / `--no-categorical` / `--range` / `--no-range`) has a structural flaw: it lets users compose flags into combinations the constraint validator must reject *after* the fact. Bad UX ÔÇö clashing flags feel like an application bug, not a user mistake.

The replacement: a single `--with=<list>` flag where the user explicitly states which constraint kinds to apply to the named fields.

**Syntax:**

```bash
# One field, one constraint kind
mdvs update reinfer status --with=categorical

# Multiple fields with the same kind (comma-separated fields)
mdvs update reinfer status,author --with=categorical

# Multiple compatible kinds on the same field(s)
mdvs update reinfer scores --with=range,length

# Strip all constraints from a field (force no auto-inference)
mdvs update reinfer priority --with=none

# Bare reinfer ÔÇö re-run defaults (heuristic-based categorical detection)
mdvs update reinfer
```

**Why `--with` and not `--as`:**

A field isn't "as" a kind ÔÇö fields *have* constraints. "Reinfer this field with range bounds" reads naturally; "reinfer this field as a range" is conceptually wrong (we're not changing the type, we're adding constraints on top).

**Validation moves into the value, not across flags:**

The user makes one explicit choice per invocation. Validation happens on the value of `--with`:

- `--with=range,categorical` Ôćĺ error at parse: "constraint kinds 'range' and 'categorical' cannot be combined on the same field"
- `--with=range,length` Ôćĺ ok (compatible)
- `--with=none,categorical` Ôćĺ error: "'none' cannot be combined with other kinds"
- `--with=foo` Ôćĺ error: "unknown constraint kind 'foo'"

Reuse the existing `ConstraintKind::conflicts_with()` for pairwise compatibility checks.

**Behavior matrix:**

| Invocation | Effect on named fields |
|---|---|
| `reinfer` (no args) | Bare reinfer ÔÇö re-runs defaults on all fields (currently: categorical heuristic) |
| `reinfer status` | Re-infer field type/paths; re-run constraint defaults |
| `reinfer status --with=categorical` | Force categorical (skip heuristic threshold) |
| `reinfer status --with=range` | Infer min/max from observed values |
| `reinfer status --with=range,length` | Infer both range and length |
| `reinfer status --with=none` | Strip all constraints; no auto-inference |

**Removed flags:** `--categorical`, `--no-categorical`, `--range`, `--no-range`. The `--max-categories` and `--min-repetition` flags stay (they're heuristic tuning, orthogonal to the kind selection).
