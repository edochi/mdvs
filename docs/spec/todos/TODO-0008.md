# TODO-0008: Support value boundary constraints on numeric fields

## Summary

Add `min` and `max` constraints for range validation during check. Builds on the constraint infrastructure from TODO-0006. Supports Integer, Float, Array(Integer), and Array(Float) fields. Bounds are inclusive. Optionally inferred via `update reinfer --range`.

## Details

### Supported types

- `Integer` — scalar, both bounds must be integers
- `Float` — scalar, bounds can be integer or float (widened to float for comparison)
- `Array(Integer)` — element-level, same rules as Integer
- `Array(Float)` — element-level, same rules as Float

For arrays, each element is checked against the bounds individually. If any element is out of range, the violation lists the offending elements (same pattern as `InvalidCategory` on arrays).

### Type matching and widening

Reuse the existing widening algorithm for comparisons:

- Integer field + Integer bound → compare as i64
- Float field + Float bound → compare as f64
- Float field + Integer bound → widen bound to f64, compare as f64

**Config error** if the field is Integer (or Array(Integer)) but a bound is Float. This catches likely mistakes — a float bound on an integer field is almost certainly wrong (e.g., `min = 0.5` on an Integer field). The user should write `min = 1` instead.

### Constraint kind

New `ConstraintKind::Range` variant on the existing enum:

```rust
enum ConstraintKind {
    Categories(Vec<toml::Value>),
    Range { min: Option<toml::Value>, max: Option<toml::Value> },
}
```

**`validate_for_type()`**: returns error for non-numeric types (Boolean, String, Object, and arrays of non-numeric types). Also returns error if a Float bound is used on an Integer field.

**`validate_value()`**: checks numeric value falls within `[min, max]` (inclusive). One or both bounds may be present. Returns `ConstraintViolation` on failure.

**`conflicts_with()`**: returns error when paired with `Categories` on the same field (redundant — if you enumerate all allowed values, a range is meaningless). Returns `None` when paired with `Length` (different domains — range constrains value, length constrains size).

### Serde layer

New fields on `Constraints` struct:

```rust
struct Constraints {
    categories: Option<Vec<toml::Value>>,
    #[serde(default, skip_serializing_if = "Option::is_none")]
    min: Option<toml::Value>,
    #[serde(default, skip_serializing_if = "Option::is_none")]
    max: Option<toml::Value>,
}
```

`active()` produces `ConstraintKind::Range` when either `min` or `max` is `Some`.

### TOML representation

```toml
# Scalar Integer with both bounds
[[fields.field]]
name = "rating"
type = "Integer"

[fields.field.constraints]
min = 1
max = 5

# Scalar Float with lower bound only
[[fields.field]]
name = "temperature"
type = "Float"

[fields.field.constraints]
min = 273.15

# Float field with integer bound (widened to float)
[[fields.field]]
name = "score"
type = "Float"

[fields.field.constraints]
min = 0
max = 100

# Array of Integers with range
[[fields.field]]
name = "ratings"
type = { array = "Integer" }

[fields.field.constraints]
min = 1
max = 10
```

### Validation

New `ViolationKind::OutOfRange`:

- **rule**: `min = 1, max = 5`
- **detail** (scalar): `got 0`
- **detail** (array): `elements out of range: [0, -1]`

### Inference

Range constraints are **not** inferred automatically by default — unlike categories, there's no good heuristic for deciding a field "should" have bounds.

However, bounds **can** be inferred on demand via `update reinfer`:

- `mdvs update reinfer <field> --range` — infer min/max from observed values across all files
- `mdvs update reinfer <field> --no-range` — strip range constraints
- The inferred min is the minimum observed value, the inferred max is the maximum observed value

This follows the same pattern as `--categorical` / `--no-categorical` for categories. Without the flag, reinfer does not touch range constraints.

### Conflicts

- `categories` + `range` on the same field → config error (redundant — enumerated values already define the range)
- `range` + `length` on the same field → no conflict (different domains)

### Config load validation

`validate_config()` (already called on mdvs.toml load) catches:

1. Range on a non-numeric field type → error
2. Float bound on an Integer field → error
3. `min > max` when both are present → error
4. Range + categories on the same field → error

### Storage/Query

No change. Constraints are validation-only.
