# TODO-0089: Warn on stale index during search

## Summary

The `search` command never checks whether the index is up to date with the files on disk. If files have been added, removed, or edited since the last `mdvs build`, search silently returns results from the stale index. Add a warning so users know when they should rebuild.

## Context

Currently, search checks:
- Index exists (fails fast if not)
- Model in config matches model in parquet metadata (hard error on mismatch)

But it does not check whether the indexed files reflect the current state of the directory. The `auto_build` config in `[update]` only affects the `update` command — search is purely read-only against whatever index exists.

## Approaches

Three levels of staleness detection, with increasing cost and accuracy:

### Option 1: File count only

Glob walk on disk to count files matching the scan pattern. Compare against `files_indexed` from parquet metadata (already available via `backend.stats()`).

- **Cost:** Very cheap — just a directory walk, no file reads
- **Catches:** Additions and deletions (count changes)
- **Misses:** Edits (same count, different content), renames (count may stay the same)
- **Implementation:** Use `ScanConfig` to walk and count, compare against `IndexStats.files_indexed`

### Option 2: Filename set comparison

Scan filenames on disk + read `read_file_index()` from parquet (already uses column projection `[0, 1, 3]`, skips the heavy `data` Struct column). Compare the two filename sets.

- **Cost:** Slightly more than option 1 — reads the file index from parquet, but it's a lightweight projection read
- **Catches:** Additions, deletions, renames
- **Misses:** Edits (filename matches but content changed)
- **Implementation:** `read_file_index()` returns `Vec<FileIndexEntry>` with `filename` and `content_hash`. Collect filenames into a `HashSet`, diff against scanned filenames. Warning message can be specific: "2 new files, 1 removed since last build"

### Option 3: Full scan + hash comparison

Scan files, compute content hashes, compare against stored hashes. Same logic as `classify_files` in build.

- **Cost:** Highest — reads all file contents, computes hashes. Same cost as the classify step in build.
- **Catches:** Everything — additions, deletions, renames, and content edits
- **Misses:** Nothing (same accuracy as build's change detection)
- **Implementation:** Reuse `classify_files()` from build, or extract it to a shared location. Warning message fully specific: "2 new, 1 edited, 1 removed since last build"

## Placement

The check should run right after the index existence check (before model loading in `search::run()`), so if the warning convinces the user to build first, they don't wait for the model to load. Output the warning to stderr.

## Warning format

```
warning: index may be stale (N new, M removed since last build — run `mdvs build`)
```

Or for option 3:
```
warning: index may be stale (N new, M edited, K removed since last build — run `mdvs build`)
```

## Files

- `src/cmd/search.rs` — add staleness check before model loading
- `src/cmd/build.rs` — potentially extract `classify_files` if using option 3
