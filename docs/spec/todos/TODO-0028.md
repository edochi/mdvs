# TODO-0028: Bare field names in --where clauses

## Summary

Let users write `--where "draft = false"` instead of `--where "data['draft'] = false"`. Frontmatter fields should be queryable as top-level column names.

## Details

### Problem

Currently, frontmatter is stored in a `data` Struct column and users must use DataFusion's bracket syntax: `data['draft'] = false`. This leaks internal storage details into the user-facing API.

### Solution

After registering the parquet files as DataFusion tables, create a **view** that promotes all frontmatter fields from the `data` Struct to top-level columns:

```sql
CREATE VIEW search_files AS
SELECT _file_id, _filename,
       data['draft'] AS draft,
       data['year'] AS year,
       data['tags'] AS tags,
       ...
FROM files
```

The field list comes from `mdvs.toml` (or from the parquet's Arrow schema for the `data` column). Internal columns use prefixed names (from TODO-0027), so there are no collisions.

The `--where` clause is then applied against this view, where bare field names resolve naturally.

### Changes

- `src/index/backend.rs` — after registering parquet tables, register a view that flattens `data` fields. Update search SQL to query the view instead of raw `files` table.
- `src/cmd/search.rs` — update help text example from `data['draft'] = false` to `draft = false`
- `src/main.rs` — update clap help text for `--where`
- `README.md` — update `--where` example
- `docs/spec/commands/search.md` — update spec

### Backward compatibility

The `data['field']` syntax would still work since the `data` column is still accessible. But docs and help text should only show bare field names.
