# TODO-0104: Expose filename as bare field in --where

## Summary

Filtering search results by file path requires using the internal column name `_filename` (e.g., `--where "_filename LIKE 'blog/%'"`). This is not user-friendly — users shouldn't need to know about the internal prefix to filter by path.

## Problem

The `files_v` view promotes frontmatter fields from the `_data` Struct to bare names, but internal columns like `_filename` are not aliased. So `--where "filename LIKE 'blog/%'"` fails with "No field named filename".

Filtering by path is a common and natural use case. It should be as easy as filtering by any frontmatter field.

## Approaches

### Option 1: Add `filename` alias to `files_v` view

Add `_filename AS filename` to the `CREATE VIEW files_v` SQL. Simple, consistent with how frontmatter fields are promoted.

Risk: if a user has a frontmatter field called `filename`, it would collide. Could check for collisions during view creation (same as `check_reserved_names` in init/update).

### Option 2: Use a more intuitive name like `path`

`_filename AS path`. Less likely to collide with frontmatter fields, and arguably more intuitive since the values are relative paths, not just filenames.

### Option 3: Add both

Alias to both `filename` and `path` for discoverability.

## Files

- `src/search.rs` — `SearchContext::new()` view creation
- `src/index/storage.rs` — `check_reserved_names()` for collision detection
