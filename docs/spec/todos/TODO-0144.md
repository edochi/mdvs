# TODO-0144: Embedded Lua scripting for user-side customization

## Summary

Embed a Lua runtime (via `mlua`) into the mdvs CLI to allow users to define custom transforms, validators, computed fields, and hooks. This turns mdvs from a fixed CLI into a programmable markdown database — the core handles parsing, Arrow, Parquet, embeddings, and SQL; Lua is the escape hatch for everything domain-specific.

## Details

**Motivation:** TODO-0009 covers built-in processors for common transforms (strip wikilinks, normalize dates, lowercase). But user needs are unpredictable — arbitrary cross-field validation, computed columns, content preprocessing, reranking. A scripting layer avoids inventing a limited DSL and gives users real expressive power. Precedent: Redis, Nginx, Neovim, WezTerm all embed Lua for exactly this reason.

**Proposed hook points (5 well-defined extension points):**

### 1. Custom validators (cross-field, conditional)
```lua
-- "published" requires "date" to be set
function validate(fields)
  if fields.published == true and not fields.date then
    return "published posts must have a date"
  end
end
```

### 2. Computed fields (derived columns in parquet, not in frontmatter)
```lua
function compute(file)
  return {
    word_count = count_words(file.body),
    reading_time = math.ceil(count_words(file.body) / 200),
    has_code = file.body:match("```") ~= nil,
  }
end
```
Queryable via `--where "word_count > 500 AND has_code = true"`.

### 3. Content transform before embedding
```lua
function pre_embed(body)
  -- strip all code blocks before embedding
  return body:gsub("```.-```", "")
end
```
Controls what the search index sees without changing source files.

### 4. Custom reranking (post-search score adjustment)
```lua
function rerank(results, query)
  for _, r in ipairs(results) do
    if r.fields.date and r.fields.date > "2026-01-01" then
      r.score = r.score * 1.2
    end
  end
  return results
end
```

### 5. Custom field extraction (beyond YAML frontmatter)
```lua
function extract(file)
  if not file.fields.title then
    return { title = file.body:match("^#%s+(.-)%n") }
  end
end
```

### 6. Export / report generation
```lua
function export(files)
  -- generate a JSON feed, RSS, tag cloud, link graph...
end
```
mdvs already has all data loaded — Lua scripts consume it in arbitrary ways.

**Design questions to resolve:**

- **Where do scripts live?** Options: `.mdvs/scripts/`, a single `mdvs.lua`, inline in `mdvs.toml`, or `[hooks]` section pointing to files.
- **Sandboxing:** restrict filesystem/network access? Lua is sandboxable but the policy needs defining.
- **Error model:** how do Lua errors surface? Warnings vs hard failures? Per-hook configuration?
- **Performance:** Lua is fast, but computed fields run per-file during build. Acceptable overhead? Benchmark needed.
- **API surface:** what exactly does Lua see? Field values, file content, config, other files? Each exposure is a contract.
- **Relationship to TODO-0009:** does this subsume built-in processors, or do both coexist (built-ins for common cases, Lua for custom)?
- **Relationship to TODO-0106 (link graph):** `extract` hook could parse wikilinks/internal links, feeding a graph.

**Crate:** `mlua` (mature, supports Lua 5.4 and LuaJIT, sandboxable).

**Mental model:** mdvs = markdown database + Lua for custom transforms, validation, computed fields, ranking. Like Redis (fast KV + Lua atomics) or Neovim (text editor + Lua config).
