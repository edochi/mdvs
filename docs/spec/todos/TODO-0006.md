# TODO-0006: Support categorical constraints on fields

## Summary

Add a `categories` constraint to fields that restricts values to a declared set of allowed values. Not a new FieldType ‚Äî categorical is a validation constraint on existing types. Applies to String, Integer, and Array (element-level) fields. Automatically inferred during init/update with manual override support.

## TOML representation

```toml
[[fields.field]]
name = "status"
type = "String"
categories = ["draft", "published", "archived"]

[[fields.field]]
name = "priority_level"
type = "Integer"
categories = [1, 2, 3]

[[fields.field]]
name = "tags"
type = { array = "String" }
categories = ["rust", "python", "go"]  # each element must be one of these
```

Presence of `categories` makes the field categorical ‚Äî no separate boolean flag needed. Absence means unconstrained.

## Applicable types

- **String** ‚Äî value must be one of the listed strings
- **Integer** ‚Äî value must be one of the listed integers
- **Array(String)** / **Array(Integer)** ‚Äî each element must be one of the listed values
- **Boolean** ‚Äî already two-valued, no need
- **Float** ‚Äî continuous, doesn't apply
- **Object** ‚Äî structural, doesn't apply

## Auto-inference

During `init` and `update --reinfer`, detect categorical fields using two conditions (both must hold):

1. **Max distinct values** ‚Äî distinct value count ‚â§ cap (e.g., ‚â§ 10‚Äì12). A field with 50 distinct values is never categorical.
2. **Minimum repetition** ‚Äî `occurrences / distinct_values ‚â• 2`. Each value appears at least twice on average. Filters out unique-per-file fields like `title`.

Examples:
- `status` with 3 values across 30 files: distinct=3 (‚â§cap), avg repetition=10 (‚â•2) ‚Üí **categorical**
- `priority` with 3 values across 6 files: distinct=3 (‚â§cap), avg repetition=2 (‚â•2) ‚Üí **categorical**
- `title` with 28 values across 30 files: avg repetition=1.07 (<2) ‚Üí **not categorical**
- `author` with 3 values across 9 files: distinct=3 (‚â§cap), avg repetition=3 (‚â•2) ‚Üí **categorical** (debatable, but user can remove)

Thresholds TBD ‚Äî exact cap and repetition minimum to be tuned during implementation.

## Manual overrides

Three mechanisms:

1. **Force categorical**: `update --reinfer <field> --categorical` ‚Äî skips heuristic, collects all distinct values as categories
2. **Force NOT categorical**: `update --reinfer <field> --no-categorical` ‚Äî skips heuristic, strips categories even if heuristic would add them
3. **Manual toml edit** ‚Äî add or remove `categories` by hand; `update` (without `--reinfer`) preserves existing categories as-is

Note: `--reinfer-all` re-runs the heuristic on everything, potentially overriding previous manual removals. Use `--reinfer <specific-field>` to preserve other fields' overrides.

## Validation (check)

New violation kind: `InvalidCategory`

- **rule**: `categories = ["draft", "published", "archived"]`
- **detail**: `got "pending"`

For arrays, validation checks each element. Detail shows the offending element(s).

Null values on categorical fields: follow existing nullable logic. If `nullable = true`, null skips the category check. If `nullable = false`, null triggers `NullNotAllowed` (existing behavior, before category check).

## Storage

No change. Categorical fields remain their underlying Arrow type (Utf8, Int64, etc.) in Parquet. The constraint is validation-only.

## Query

No change. `--where` clauses on categorical fields work the same as any String/Integer field.

## Files

- `src/schema/config.rs` ‚Äî add `categories` to `TomlField`
- `src/discover/infer.rs` ‚Äî collect distinct values, apply heuristic, populate categories
- `src/cmd/check.rs` ‚Äî validate values against categories, new `InvalidCategory` violation
- `src/output.rs` ‚Äî add `InvalidCategory` to `ViolationKind`
- `src/cmd/update.rs` ‚Äî pass `--categorical` / `--no-categorical` flags, handle reinfer
