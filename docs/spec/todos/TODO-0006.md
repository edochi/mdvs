# TODO-0006: Support categorical constraints on fields

## Summary

Add a `categories` constraint to fields that restricts values to a declared set of allowed values. Not a new FieldType â€” categorical is a validation constraint on existing types. Applies to String, Integer, and Array (element-level) fields. Automatically inferred during init/update with manual override support.

## TOML representation

Constraints live in a `[fields.field.constraints]` sub-table, not as top-level field attributes. This keeps constraints visually and semantically grouped as refinements of the type, separate from structural attributes like `name`, `type`, `nullable`, `allowed`, `required`.

```toml
[[fields.field]]
name = "status"
type = "String"

[fields.field.constraints]
categories = ["draft", "published", "archived"]

[[fields.field]]
name = "priority_level"
type = "Integer"

[fields.field.constraints]
categories = [1, 2, 3]

[[fields.field]]
name = "tags"
type = { array = "String" }

[fields.field.constraints]
categories = ["rust", "python", "go"]  # each element must be one of these

[[fields.field]]
name = "title"
type = "String"
# no constraints section â€” unconstrained
```

Presence of `categories` makes the field categorical â€” no separate boolean flag needed. Absence of the `[fields.field.constraints]` section (or an absent `categories` key) means unconstrained.

### Mutual exclusion with other constraint kinds

The `[fields.field.constraints]` sub-table is shared with future constraint kinds (TODO-0008: `min`/`max`, TODO-0010: `min_length`/`max_length`). **At most one constraint kind per field.** Config validation errors if multiple constraint kinds coexist in the same `[fields.field.constraints]` section.

Constraint kinds:
- `categories` â€” enumerated allowed values (this TODO)
- `min` / `max` â€” numeric range (TODO-0008, one or both)
- `min_length` / `max_length` â€” length bounds (TODO-0010, one or both)

In practice, type restrictions already prevent most combinations (`min`/`max` only on numeric, `min_length`/`max_length` only on String/Array). The real overlaps are Integer (`categories` vs `min`/`max`) and Array (`categories` vs `min_length`/`max_length`). Mutual exclusion keeps things simple â€” can be relaxed later if a real use case emerges.

## Applicable types

- **String** â€” value must be one of the listed strings
- **Integer** â€” value must be one of the listed integers
- **Array(String)** / **Array(Integer)** â€” each element must be one of the listed values
- **Boolean** â€” already two-valued, no need
- **Float** â€” continuous, doesn't apply
- **Object** â€” structural, doesn't apply

## Auto-inference

During `init` and `update --reinfer`, detect categorical fields using two conditions (both must hold):

1. **Max distinct values** â€” distinct value count â‰¤ cap (e.g., â‰¤ 10â€“12). A field with 50 distinct values is never categorical.
2. **Minimum repetition** â€” `occurrences / distinct_values â‰Ą 2`. Each value appears at least twice on average. Filters out unique-per-file fields like `title`.

Examples:
- `status` with 3 values across 30 files: distinct=3 (â‰¤cap), avg repetition=10 (â‰Ą2) â†’ **categorical**
- `priority` with 3 values across 6 files: distinct=3 (â‰¤cap), avg repetition=2 (â‰Ą2) â†’ **categorical**
- `title` with 28 values across 30 files: avg repetition=1.07 (<2) â†’ **not categorical**
- `author` with 3 values across 9 files: distinct=3 (â‰¤cap), avg repetition=3 (â‰Ą2) â†’ **categorical** (debatable, but user can remove)

Thresholds TBD â€” exact cap and repetition minimum to be tuned during implementation.

## Manual overrides

Three mechanisms:

1. **Force categorical**: `update --reinfer <field> --categorical` â€” skips heuristic, collects all distinct values as categories
2. **Force NOT categorical**: `update --reinfer <field> --no-categorical` â€” skips heuristic, strips categories even if heuristic would add them
3. **Manual toml edit** â€” add or remove `categories` by hand; `update` (without `--reinfer`) preserves existing categories as-is

Note: `--reinfer-all` re-runs the heuristic on everything, potentially overriding previous manual removals. Use `--reinfer <specific-field>` to preserve other fields' overrides.

## Validation (check)

New violation kind: `InvalidCategory`

- **rule**: `categories = ["draft", "published", "archived"]`
- **detail**: `got "pending"`

For arrays, validation checks each element. Detail shows the offending element(s).

Null values on categorical fields: follow existing nullable logic. If `nullable = true`, null skips the category check. If `nullable = false`, null triggers `NullNotAllowed` (existing behavior, before category check).

## Storage

No change. Categorical fields remain their underlying Arrow type (Utf8, Int64, etc.) in Parquet. The constraint is validation-only.

## Query

No change. `--where` clauses on categorical fields work the same as any String/Integer field.
