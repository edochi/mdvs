# TODO-0171: Similarity-edge graph — top-K semantic neighbors per file

## Summary

Augment the link graph from [TODO-0106](TODO-0106.md) with a parallel **implicit edge** source: for every file, materialize the top-K most cosine-similar other files using the existing vector index. Stored as a denormalized column on the per-chunk Lance dataset, queryable via `--where`, and composable with the explicit link graph in a unified "related notes" view. The high-value capability is surfacing connections that no one wrote down explicitly — semantic neighbors a reader didn't bother to wikilink but a recommender system would.

## Motivation

The explicit-link graph in TODO-0106 captures only the structure a writer remembered to encode. Real notebooks are full of related material that's never linked — different vocabulary for the same topic, parallel projects, shared references — and surfacing these is the main thing a search-aware notes tool can offer that hand-curated graphs cannot.

The existing semantic search machinery already computes the answer for one query at a time. Pre-computing the answer for every file as a fixed top-K neighborhood turns "related notes" from a query into a column read, and makes it possible to compose semantic adjacency with link-graph operations.

## What gets stored

A single denormalized column on each chunk row of `.mdvs/index.lance/`:

```
similar_files: List<Struct<
    file_id: Utf8,
    score:   Float32,    // cosine similarity, range [0, 1]
>>
```

- File-level (not chunk-level): for every chunk row of file Y, the column carries the same list — the top-K files most similar to file Y. Lance's columnar encoding RLE-compresses the duplication within a file's chunks (same trick as `filepath`, `links_out_file`, etc.; see TODO-0016 storage rationale).
- K is config-driven (`[graph].similar_top_k`, default 10). Storage cost per file is K small structs ≈ a few hundred bytes; for a 10K-file vault, ~tens of MB total — trivial alongside embeddings.
- Score is included so consumers can threshold ("only neighbors above 0.7"), weight ranking, or sort. Without the score the column would be much less useful.

### Why not a separate edge table

Same architectural argument as TODO-0106. A two-dataset shape (`files.lance` with explicit edges, `similarity.lance` with implicit edges, joined at search time) re-introduces the LanceDB no-JOIN friction that TODO-0016 deliberately removed. Stays single-table, denormalized.

### File granularity vs chunk granularity

File-level neighbors are the right grain for the user-facing feature ("related notes to this note"). Chunk-level neighbors are more granular but mostly noise — adjacent chunks of the same file dominate, and the deduplication step we already run for `search` would have to run for graph queries too. File granularity is computed by:

1. For each file, take its chunks' embeddings.
2. Either (a) average them into a file vector and ANN that, or (b) ANN each chunk and aggregate the per-chunk neighbors into file-level results (best-chunk-per-file, as `search` already does, with the OVER_FETCH_FACTOR pattern).

Option (b) reuses existing dedupe logic and gives better recall (the file vector loses chunk-specific signal). Lean (b) as the default; (a) might be worth offering as a faster opt-in for very large corpora.

## How it's computed at build time

The existing vector index (cosine, `nearest_to`) already supports the query. The build-time pass is:

1. After the chunk Lance table is written and the vector index (or exact flat scan path) is available, iterate every chunk row.
2. For each chunk, run `nearest_to(chunk.embedding).limit(K * OVER_FETCH_FACTOR)`.
3. Filter out same-file hits; group by `file_id`; keep the max chunk-similarity per file (same logic `search` uses).
4. Take top K, store as the `similar_files` list on every chunk row of the source file.

Build-time cost is O(N_chunks × K_search), which is exactly what `search` pays per query — paid once at build time instead of per query. For a 10K-chunk corpus with K=10 and the IVF-PQ index active (above `VECTOR_INDEX_MIN_ROWS = 10_000`), this is comfortably sub-minute. For smaller corpora on the flat-scan path it's O(N²) but N is small (<10K), still seconds.

Incremental story (see TODO-0170): when a file's body changes, its outgoing similarity edges change, but also potentially its *incoming* similarity edges from other files (because the changed file might now be in or out of their top-K). For v0, accept that the similarity pass is a corpus-wide pass triggered when:
- the embedding model / revision changed (forces rebuild anyway), or
- more than some threshold of files changed in a build (e.g., >1%), or
- the user passes `--rebuild-similarity`.

For small per-build deltas, leave the column stale; document the staleness and offer the explicit recompute flag. This is acceptable because similarity edges are a soft signal — staleness degrades quality but doesn't break correctness, unlike `links_in_file` which has correctness implications.

## Query model

### 1-hop, Lance-native

```bash
# files most semantically similar to a specific file
mdvs graph similar <file>                       # reads the column directly

# search filtered by being a semantic neighbor of X
mdvs search "calibration" --where "array_has(similar_files.file_id, '<file>')"

# search ranked by similarity to X (composable with semantic score)
mdvs search "calibration" --boost-by similar_files
```

The `array_has(similar_files.file_id, ...)` form works because LanceDB / DataFusion supports struct field accessors inside `array_has`. The translator already handles struct-field access for `data.*` paths; the same machinery extends to this column.

### Composed with explicit links (TODO-0106)

The whole point. A `mdvs graph related <file>` view merges:

- Files in `links_out_file` (this file links to them)
- Files in `links_in_file` (they link to this file)
- Files in `similar_files` (they're semantically close)

Combined ranking might weight these (e.g., `1.0 * link_present + 0.7 * similarity_score`), or just bucket them ("explicit links" / "semantic neighbors") in the output. UX decision; design separately when implementing.

## What this explicitly does NOT do

Recording the alternatives considered and rejected so the design boundary is clear:

- **No `community: Int32` label per file.** Hard-partition graph clustering (Leiden / Louvain) is the standard algorithm but the wrong output shape for a notes corpus: notes naturally belong to multiple topics simultaneously; partition labels are unstable across incremental builds (renumbering, non-determinism); and the per-node similarity neighborhood we already materialize carries strictly more information than any single label. If a clustered overview view becomes valuable later (e.g., a `mdvs graph overview` map), compute the clustering at query time over the materialized similarity + link graph in petgraph and discard the labels — same pattern TODO-0106 uses for traversal.
- **No frontmatter co-occurrence as a stored edge type.** "Both files have `topic = X`" is already expressible via `--where` and doesn't need materialization. Worth a recipe in the docs; not worth a column.
- **No named-entity / concept extraction.** Would require either an NER ONNX model in the build path or LLM calls at index time. Changes the project's character (build no longer offline / model-free at the validation layer; binary size grows; index time multiplies). Out of scope for v0; revisit once the simpler implicit-edge work has been used in anger.
- **No co-citation edges as a stored type.** "A and B both link to C" is a derived query over `links_out_file` (self-join or DataFusion equivalent) and doesn't need its own column. Surface it in the `graph` command output instead.

## Commands

- **`build`** — add the similarity pass after the vector index is ready; controlled by `[graph].similar_top_k` (default 10) and `[graph].similar_min_score` (default 0.0, no threshold).
- **`search`** — `--where "array_has(similar_files.file_id, '<file>')"` works out of the box once the column lands.
- **New `graph similar <file>`** subcommand — direct column read, no petgraph, returns the K neighbors with scores.
- **New `graph related <file>`** subcommand — merges explicit links (TODO-0106) and similarity edges, ranked.
- **`info`** — surface similarity stats (avg neighbor score, files with no neighbors above threshold).

## Open design questions

1. **Default K.** 10 is the obvious starting point; could be lower (5) to keep the column tiny, or higher (20) to give downstream consumers room to threshold. Pick after the storage cost is measured on the K8s corpus.
2. **Similarity floor.** A file with no truly close neighbors will pull in noise at K=10. Optional `[graph].similar_min_score` truncates the list; what's a sensible default? Probably 0 (no truncation) for v0; let users opt in.
3. **Self-similarity handling.** The same-file filter must run *after* the ANN call (the index doesn't know to skip same-file matches). Easy at the chunk level (`chunk.file_id != current.file_id`); explicit in the algorithm.
4. **Stability across embedding model swaps.** Changing the model invalidates the entire `similar_files` column. Schema-hash mechanism already covers this — model identity change triggers full rebuild. Document the implication.
5. **Symmetry.** Cosine is symmetric, but top-K isn't: file A in B's top-10 doesn't imply B in A's top-10. Worth exposing both directions? "Files I'm similar to" vs "files similar to me" reads the same column and produces the same result; "files for which I'm in their top-K" requires a reverse-index pass (analogous to `links_in_file` materialization in TODO-0106). For v0, ship only the forward direction; reverse is a follow-up if anyone asks.
6. **Interaction with hybrid / FTS modes.** Similarity is purely vector-based. Should we also materialize FTS-based "lexically similar" neighbors? Probably not — BM25 over short text doesn't carry the same "related concept" semantics, and the marginal value seems low. Skip unless evidence emerges.

## Scope and rollout

1. **Slice 1 — column + build pass.** Add the `similar_files` column to the Arrow schema; implement the build-time similarity pass; gate it on a config flag (`[graph].similar = true`, default off initially to ease rollout). Expose `mdvs graph similar <file>`.
2. **Slice 2 — incremental policy.** Implement the "rebuild on model change / large delta, manual flag otherwise" policy. Surface staleness in `info`.
3. **Slice 3 — unified related view.** Once TODO-0106 has shipped slice 1, build `mdvs graph related <file>` that merges both edge sources.
4. **Slice 4 (optional) — boost-by ranking.** `mdvs search ... --boost-by similar_files <file>` to bias results toward semantic neighbors of a reference file.

Slice 1 alone is the minimum that proves the design works. Default the build flag to off until the storage cost and quality have been measured on a real corpus (Refractions, or the K8s benchmark corpus).
