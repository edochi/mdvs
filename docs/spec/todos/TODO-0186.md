# TODO-0186: `mdvs explain` ŌĆö proactive schema query

## Problem

Today an agent that wants to write a new note has two options:

1. **Guess + retry.** Compose frontmatter, run `mdvs check`, parse
   violations, fix, repeat. Reactive.
2. **Read `mdvs.toml` and reverse-engineer scope.** Parse the TOML,
   evaluate every field's `allowed` / `required` globs against the
   target path, derive the applicable rules. Doable but it requires
   the agent (or the harness writing the agent prompt) to reimplement
   mdvs's path-scoping semantics. Brittle.

Neither is what an agent harness actually wants. The shape that
matches the use case is: "I'm about to write `notes/projects/alpha/
draft.md` ŌĆö what fields are allowed / required at that path, and
what constraints govern each?" One call, structured answer, agent
generates valid frontmatter on the first try.

This pairs with TODO-0177's audience reframe (LLM-curated KBs) and
makes the agent-callable surface concrete. It's the proactive twin
of `mdvs check`.

## Goal

New subcommand:

```
mdvs explain <file_path> [--output pretty|markdown|json]
```

`<file_path>` is the candidate file path (need not exist). Output
reports the fields whose `allowed` glob matches `<file_path>`, with
their type, nullability, constraints, preprocess pipeline, and
whether they are `required` at that path. The response also reports
fields known to mdvs.toml that are *not* applicable here (so the
agent knows what NOT to write), and a textual `scope_note` for
edge cases.

`--output pretty` is human-readable (box-drawing tables);
`--output markdown` is the same content as GFM tables, suitable for
piping or agent consumption; `--output json` is the strict structured
contract for `jq` or programmatic consumption.

### `<file_path>` must be a file path, not a directory

`explain` answers a per-file question. Directories don't map to
globs cleanly. If `<file_path>` is a directory:

```
error: `mdvs explain` requires a file path. For a directory-tree
overview of which rules apply where, use `mdvs explain --tree <dir>`
(see TODO-0186-tree-followup).
```

A directory-tree overview is a different operation (walk the tree,
group fields by glob bucket, show schema landscape). Capture it as
a follow-up TODO if agent users actually ask for it; the per-file
form is what the proactive-write loop needs.

## Proposed JSON shape

```json
{
  "path": "notes/projects/alpha/draft.md",
  "vault": "/Users/x/notes",
  "frontmatter_format": "auto",
  "applicable_fields": [
    {
      "name": "title",
      "type": "String",
      "required": true,
      "nullable": false,
      "constraints": null,
      "preprocess": []
    },
    {
      "name": "status",
      "type": "String",
      "required": false,
      "nullable": false,
      "constraints": { "categories": ["draft", "review", "published"] },
      "preprocess": []
    },
    {
      "name": "calibration.baseline.wavelength",
      "type": "Float",
      "required": false,
      "nullable": false,
      "constraints": { "range": { "min": 400.0, "max": 1100.0 } },
      "preprocess": ["widen_int_to_float"]
    }
  ],
  "disallowed_fields": [
    "firmware_version",
    "wavelength_nm",
    "equipment_id"
  ],
  "scope_note": "3 fields apply at this path. 3 other fields exist in mdvs.toml but their `allowed` globs do not cover this path ŌĆö using them here would trigger a `Disallowed` violation. Frontmatter keys not in either list are treated as 'new fields' (informational, surfaced by `mdvs update`)."
}
```

### Three categories, by intent

The split is deliberate. Each category answers a different question
the agent is implicitly asking:

| Category | What it tells the agent | Cost in JSON |
|---|---|---|
| `applicable_fields` | What you should USE ŌĆö full schema with type, constraints, required, preprocess. | Full schema, ~50ŌĆō200 bytes per field. |
| `disallowed_fields` | What you must NOT use here unless you first change `allowed` in mdvs.toml. Names only ŌĆö no full schema needed; the agent can call `mdvs explain` on the field's typical path if it wants the details. | Just a string list, ~10ŌĆō20 bytes per name. |
| Implicit: "new fields" | Anything not in either list is a never-before-seen field. mdvs will report it as informational (not a violation); `mdvs update` would later add it to the schema. | Not in JSON at all ŌĆö the agent infers this from "not present in either list". |

`scope_note` carries the textual hint that bridges the three
categories ŌĆö especially important for the edge case where
`applicable_fields` is empty (e.g., the agent created a new folder
at the vault root and is writing the first note there). The note
makes it explicit: "no rules apply yet; write whatever, then run
`mdvs update` to bring the schema in line ŌĆö or set `allowed = ["**"]`
on the fields you want to apply globally."

## Approach

1. **Resolve vault.** Walk up from `<file_path>` looking for
   `mdvs.toml`; error cleanly if none found.
2. **Load schema.** Reuse the existing `MdvsToml::read()` path.
3. **Partition fields.** For each `[[fields.field]]`, check whether
   `<file_path>` (relative to the vault root) matches any `allowed`
   glob.
   - **Match ŌåÆ `applicable_fields`** with full schema. Mark
     `required: true` if `<file_path>` also matches any of its
     `required` globs.
   - **No match ŌåÆ `disallowed_fields`** (name only).
4. **Compose `scope_note`.** A short textual summary keyed on the
   sizes of the two categories (and whether the field is in a "bare
   files allowed" path).
5. **Render output.** Text via the existing `Block` / `Render`
   infrastructure (probably a small table); JSON via a new
   `outcome::commands::ExplainOutcome` struct + serde.

The path-scoping logic is already implemented in
`schema/constraints/` and used by `cmd::check::validate`. Reuse it
exactly ŌĆö no parallel re-implementation.

## Design questions to settle before implementation

1. **Command name** ŌĆö `explain` chosen. Reads as an agent verb,
   short, unambiguous. Alternative `fields <path>` is more literal
   but loses the "tell me about" flavor. Keeping `explain`.
2. **Canonical JSON Schema slice** ŌĆö should `--canonical` (or
   `--json-schema`) flag emit the JSON Schema 2020-12 subset for
   just the applicable fields? Useful for an agent feeding the
   schema into a structured-output generator. Defer to v2 unless an
   early caller needs it.
3. **Example skeleton** ŌĆö should the JSON include an
   `example_skeleton` field (ready-to-paste frontmatter block with
   required fields and sensible defaults)? Helpful for "write me a
   new note" agent flows but adds template logic. Decision: NO in
   v1. Add `--with-skeleton` flag in v1.5 if asked for.
4. **Path existence** ŌĆö `<file_path>` is treated as a candidate
   path; the file need not exist. (Confirmed: this is the whole
   point ŌĆö pre-write query.)
5. **Empty `applicable_fields` + non-empty `disallowed_fields`** ŌĆö
   the agent might be in a new sub-tree where nothing applies but
   the vault has other rules elsewhere. `scope_note` makes the
   situation legible. The agent's three actions in this case:
   - Write any frontmatter ŌĆö none of it will fail validation.
   - Or pick a field from `disallowed_fields` and first edit
     `mdvs.toml` to extend its `allowed` glob to cover this path.
   - Or skip frontmatter entirely (bare file).

## Verification

- `mdvs explain` on a path inside `example_kb/` returns a sensible
  partition ŌĆö e.g., `example_kb/projects/alpha/notes/foo.md` should
  show `author`, `status`, `attendees`, etc. in
  `applicable_fields`, with `firmware_version` and
  `wavelength_nm` in `disallowed_fields`.
- `mdvs explain example_kb/some-new-area/note.md` (a path covered
  by NO existing `allowed` globs) returns
  `applicable_fields: []` plus the right `scope_note`.
- `mdvs explain example_kb/` (a directory path) errors with the
  pointer to the tree-overview follow-up.
- `--output json` is `jq`-parseable and stable across releases.
- New integration test in `crates/mdvs/tests/` confirms the
  filtering matches the same predicate that `cmd::check::validate`
  uses (no parallel implementation drift).
- `cargo test --features testing-mocks` + clippy + fmt clean.
- Help text on the new subcommand is concise; example in book.

## Out of scope

- Live validation against proposed frontmatter content (i.e.,
  "here's my draft frontmatter, tell me what's wrong"). That's
  what [TODO-0188](TODO-0188.md) (`mdvs check --stdin`) covers.
- Multiple paths in one call. `for f in files; do mdvs explain $f
  --output json; done` handles batch usage; a `--paths-from-stdin`
  flag is a v2.
- Directory-tree overview (`mdvs explain --tree <dir>`). Captured
  as a follow-up TODO if agent users ask for it.

## Impact

Closes the gap between "we have a schema" and "an agent can use the
schema." Without this, the agent loop is writeŌåÆcheckŌåÆfixŌåÆrepeat;
with it, the agent gets the rules upfront and writes valid
frontmatter on the first try. Direct enabler for [TODO-0187]
(agent harness hook recipe) ŌĆö especially its pre-explain mode.
