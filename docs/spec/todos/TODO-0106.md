# TODO-0106: Link graph from internal links and wikilinks

## Summary

Extract internal links (markdown `[text](path.md)`, GitHub-style relative links) and wikilinks (Obsidian-style `[[page]]` / `[[page|alias]]`) from markdown content to build a link graph between files. Store edges in the existing Parquet storage, enabling graph-aware search and navigation — all from markdowns as sources and Parquet as storage, no external services.

## Details

### What to extract

Two link formats:

1. **Markdown links**: `[text](relative/path.md)` — standard internal links used in GitHub repos, mdBook, etc.
2. **Wikilinks**: `[[page]]`, `[[page|display text]]`, `[[page#heading]]` — Obsidian and wiki-style links

Both resolve to file paths within the scanned directory. External URLs (http/https) are ignored.

### Storage

Store link edges in the existing Parquet files. Options:

- **New column in `files.parquet`**: a `_links` column (Array of Strings) listing resolved target file paths. Simple, queryable via `array_has(_links, 'path/to/target.md')`.
- **New `links.parquet`**: separate edge table with `source_file_id`, `target_file_id`, `link_text`, `link_type` (markdown/wikilink), `line_number`. More granular, supports richer graph queries.

### Graph capabilities

With link edges stored, several features become possible:

- **Search + graph context**: "find notes related to X that link to Y" — combine semantic similarity with graph connectivity
- **`--where` on links**: `--where "array_has(_links, 'projects/alpha/overview.md')"` — find files that link to a specific page
- **Backlinks**: reverse lookup — which files link to this file?
- **Graph traversal**: N-hop neighbors, connected components, path finding between notes
- **Orphan detection**: files with no incoming or outgoing links
- **Link validation**: detect broken internal links (target doesn't exist)

### Commands

- **`build`**: extract links during the scan/chunk phase, store in Parquet alongside embeddings
- **`search`**: filter by link presence via `--where`
- **`check`**: optionally validate that link targets exist (broken link detection)
- **`info`**: show link stats (total links, orphans, most-linked files)
- **New `graph` command?**: dedicated graph queries — backlinks, neighbors, shortest path

### Design questions

- Should link extraction happen during scan (cheap, always available) or build (only when index exists)?
- Should wikilink resolution be configurable (case-insensitive matching, folder search)?
- How to handle ambiguous wikilinks (multiple files with the same name in different directories)?
- Should backlinks be materialized in Parquet or computed at query time via DataFusion JOIN?
- Is a separate `graph` command warranted, or should graph queries be expressible via `--where` on `search`?

### Scope

This is a significant feature. Start with:
1. Extract links during scan
2. Store as `_links` Array column in `files.parquet`
3. Make queryable via `--where "array_has(_links, 'path')"`
4. Backlinks via DataFusion self-join

Advanced graph features (traversal, visualization, dedicated command) can follow incrementally.
