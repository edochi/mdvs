# TODO-0106: Link graph from internal links and wikilinks

## Summary

Extract internal links (markdown `[text](path.md)`, GitHub-style relative links) and wikilinks (Obsidian-style `[[page]]` / `[[page|alias]]` / `[[page#heading]]`) from markdown content to build a directed link graph between files. Store edges as denormalized columns on the existing per-chunk Lance dataset (consistent with the TODO-0016 single-table decision) and use [`petgraph`](https://docs.rs/petgraph) as a query-time scratchpad for traversal. The killer capability is composing a graph filter with semantic / FTS / hybrid search in a **single LanceDB query** — e.g., "chunks semantically similar to X that also link to Y".

## What to extract

Two link formats, both resolved to file paths within the scanned root; external `http(s)://` links are ignored.

1. **Markdown links**: `[text](relative/path.md)` — standard GitHub / mdBook style.
2. **Wikilinks**: `[[page]]`, `[[page|display text]]`, `[[page#heading]]` — Obsidian and wiki style.

Resolution rules to decide on:
- Wikilinks resolve relative to the scan root, with a configurable folder-search fallback (Obsidian's "shortest path when unambiguous" mode).
- Ambiguous wikilinks (`[[overview]]` with multiple `overview.md` files): surface as a `LinkAmbiguous` violation in `check` by default; allow a `[graph].resolve = "proximity"` knob to silently prefer the closest match by directory distance.
- Broken links (target not in the file set): surface as a `LinkBroken` violation in `check`. Soft by default (warning), strict opt-in via `[graph].strict = true`.

## Storage

Stays single-table (Lance dataset at `.mdvs/index.lance/`). The TODO-0016 Lance swap deliberately collapsed the prior `files.parquet` + `chunks.parquet` shape into one denormalized row-per-chunk table to avoid LanceDB's lack of a JOIN primitive in its retrieval API. Graph fields follow the same denormalization principle:

```
links_out_chunk: List<Utf8>      // edges resolved from THIS chunk's text
links_out_file:  List<Utf8>      // file-level out-edges (union over file's chunks), denormalized per row
links_in_file:   List<Utf8>      // file-level in-edges, materialized at build time by inversion
pagerank:        Float32         // optional, per-file, denormalized per row (future)
```

All four are top-level Arrow columns on the existing chunk row, joining `chunk_id`, `filepath`, `chunk_text`, `embedding`, `data`, etc. Lance's columnar encoding run-length-compresses the file-level columns across chunks belonging to the same file, so the duplication cost is near-zero in practice (same compression that makes today's per-chunk `filepath` and `data` duplication negligible).

### Alternatives considered and rejected

- **Reintroduce a separate `files.lance` dataset (revert the TODO-0016 collapse).** Reads cleaner for pure file-level queries, but combining a vector / FTS / hybrid search with a graph filter requires a DataFusion-level join across the two datasets — exactly the friction TODO-0016 deliberately removed. The LanceDB hybrid reranker and `--where` translator don't compose across tables. Also doubles the incremental-cache (TODO-0170) state to track.
- **One row per file with chunks as a nested `List<Struct<embedding: FixedSizeList<...>, chunk_text, ...>>`.** Arrow / Lance support the storage shape, but LanceDB's vector ANN, BM25 FTS, and hybrid indexes all require their target column to be **top-level** on the row, not nested inside a List or Struct. Nested embeddings would break vector search entirely; nested `chunk_text` would break FTS. Multi-vector / late-interaction retrieval (ColBERT-style) is the use case that motivates nested-vector schemas, but it's not a production feature in `lancedb` 0.29 and requires a different (contextual) embedding model than Model2Vec.
- **Per-chunk `links_out` only, file-level edges always derived at query time.** Workable, but every graph-filtered search becomes a `GROUP BY file_id` aggregate. Materializing `links_out_file` / `links_in_file` denormalized onto chunk rows lets the same `array_has(...)` filter that powers `--where` today serve graph queries with zero aggregation cost.

## Query model

### 1-hop, fully Lance-native (the common case)

```bash
# files that link OUT to a target
mdvs search "calibration drift" --where "array_has(links_out_file, 'reference/protocols/spec-200x.md')"

# files that link IN (backlinks)
mdvs search "humidity" --where "array_has(links_in_file, 'projects/alpha/overview.md')"

# orphans
mdvs search "" --where "cardinality(links_in_file) = 0 AND cardinality(links_out_file) = 0"
```

All three are single pushed-down filters on the existing Lance dataset. The `--where` translator (already handling `array_has` for other Array fields) extends to these columns trivially.

### Multi-hop and graph algorithms (rare; via petgraph)

For BFS / DFS / shortest-path / connected components / PageRank, project only `file_id` + `links_out_file` from Lance into a `petgraph::DiGraph` in memory:

- Column projection: Lance reads two small string columns, skipping the embedding column (which is ~99% of the bytes).
- Dedupe by `file_id` while building (chunk rows for the same file carry identical `links_out_file` thanks to denormalization).
- Vault scale: 10K files × 30 edges ≈ 300K edges ≈ a few MB in memory. Build time milliseconds.
- Bounded BFS from a starting node can also be done iteratively with predicate pushdown (`WHERE file_id IN (frontier)` per hop), avoiding the full graph load for shallow traversals.

PageRank is the sleeper feature: compute offline at build time, store as a `pagerank: Float32` column denormalized per chunk row, then weight search results by `score * pagerank`. Better ranking than Obsidian's graph view, no query-time cost.

## Commands

- **`build`** — extract links during the existing scan/chunk pipeline; populate `links_out_chunk`; do an inversion pass to materialize `links_out_file` + `links_in_file`; optionally compute `pagerank`.
- **`check`** — emit `LinkAmbiguous` and `LinkBroken` violations alongside frontmatter violations. Cheap (no model needed); fits the existing model-free validation layer.
- **`search`** — graph filters via `--where array_has(links_in_file, ...)` etc.; works in all three modes (semantic / fulltext / hybrid) for free.
- **`info`** — surface link stats (total edges, orphan count, top-N most-linked files, broken-link count).
- **New `graph` subcommand** — dedicated for traversal-style queries:
  - `mdvs graph backlinks <file>` — direct column read of `links_in_file` (no petgraph)
  - `mdvs graph neighbors <file> --depth N` — bounded BFS, iterative pushdown for small N or full-graph petgraph for large N
  - `mdvs graph orphans` — column filter
  - `mdvs graph broken` — `links_out_chunk` entries not in the file_id set
  - `mdvs graph dot [--root <file>]` — emit GraphViz DOT for visualization (no rendering dep)

## Open design questions

1. **Per-chunk vs file-level link granularity for backlink UX.** `links_out_chunk` enables "Y links to you in chunk 3 at line 47" precision; the file-level columns lose that. Keep both? `links_out_chunk` is cheap (just stores what's already extracted; no inversion needed).
2. **Wikilink resolution mode default.** Strict (ambiguity = violation) vs proximity-prefer (Obsidian default). Lean strict — users with messy vaults can opt into proximity.
3. **Where to detect cycles for `--strict` mode.** During build (block index write) or during `check` only?
4. **Should `links_in_file` materialization respect path-scoped fields?** Currently links cross any directory; a future per-glob link scope is conceivable but probably YAGNI for v0.
5. **Storage shape for richer backlink metadata.** `Array<Utf8>` (just target paths) vs `Array<Struct<target, line, anchor_text>>` for "X mentioned this in chunk Z at line N" UX. The struct shape works in Lance (`array_has` on the struct field via DataFusion struct accessor); cost is more storage. Decide after the simple form ships.
6. **`graph` command vs `search --graph`.** A dedicated command reads better; folding traversal into `search` overloads it. Lean dedicated command.

## Interaction with other TODOs

- **TODO-0170 (incremental cache).** `links_in_file` is a global derived column — editing one file's outgoing links changes other files' backlinks. The incremental cache needs to track reverse dependencies, or accept that the link-inversion pass is always corpus-wide (still cheap — it's a single scan of two small columns and a hash-map invert). Probably the simpler answer: link-index rebuild is a separate, always-full pass that's fast enough to run unconditionally. Coordinate the decision with 0170 before implementing.
- **TODO-0156 (Array of structured items).** If we choose `Array<Struct<target, line, text>>` for the rich backlink shape, this overlaps 0156's "Array of structured items without violating Wave C flattening" question. Internal storage (synthetic column not exposed via `[[fields.field]]`) sidesteps the user-facing concern.
- **TODO-0016 (Lance swap, done).** The denormalization decision here is a direct application of 0016's "single table, file fields denormalized onto chunk rows" architecture. This TODO is consistent with that choice and does not revisit it.
- **TODO-0157 (incremental ANN optimize, open).** Independent — link columns don't participate in the vector index.
- **TODO-0171 (similarity-edge graph).** Companion implicit-edge source. Same denormalization principle, same `graph` subcommand surface, same `[graph]` config section. The `mdvs graph related <file>` view merges link edges from this TODO with similarity edges from 0171. Design them coherently.

## Scope and rollout

Stage in shippable slices:

1. **Slice 1 — extract + store.** Add `links_out_chunk`, `links_out_file`, `links_in_file` columns; populate during build; surface `LinkAmbiguous` / `LinkBroken` in `check`. No new commands. Query-time access via `--where array_has(...)`. This alone delivers backlinks and graph-filtered search.
2. **Slice 2 — `graph` subcommand.** Backlinks, neighbors (depth-bounded), orphans, broken-link report. Pulls in `petgraph` as a dependency.
3. **Slice 3 — PageRank as ranking signal.** Compute at build time, denormalize per row, expose via `search --rank-by score*pagerank` (or always-on weighting, opt-out).
4. **Slice 4 (optional) — DOT export and richer backlink struct.** Visualization handoff, line-number / anchor-text backlinks.

Slices 1 + 2 cover ~95% of the user-facing graph value and are tractable as one or two PRs. Slice 3 is independent and can wait. Slice 4 is nice-to-have.
