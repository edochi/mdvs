# TODO-0106: Link graph from internal links and wikilinks

## Summary

Extract the links a markdown corpus already contains — markdown `[text](path.md)`, Obsidian wikilinks `[[page]]` / `[[page|alias]]` / `[[page#heading]]`, and frontmatter references (`parent = "[[index]]"` via `is_reference = true`) — into a queryable relationship layer, and fold it into the commands mdvs already has: `search` (relationship filters), `check` (broken / ambiguous links), and `info` (stats). No new top-level command, no separate dataset. Explicit links are stored as a column on the chunk row; semantic "related notes" are computed on demand from the embeddings mdvs already stores; petgraph handles traversal at query time.

## Scope — extracted links only

This TODO covers links **extracted from markdown content** — connections *derivable from the files themselves* (a wikilink in prose, an `is_reference` frontmatter value). They are build state: recreatable, gitignored, never a source of truth.

**Out of scope:** *declared*, out-of-band references — e.g. a link a user or agent authors between a doc and a specific code symbol, maintained in a committed sidecar with drift / staleness detection over code. That is a separate concern (a distinct traceability project), not mdvs, and it must not pull mdvs toward a code-indexing or committed-graph model. The dividing line is **extracted (mdvs) vs declared (not mdvs).**

## Storage — a `links` column on the chunk row (no `edges.lance`)

A separate persisted edge dataset was considered and dropped. Persisting `.mdvs/` state is justified by *recompute cost* (embeddings need model inference), and link extraction is a cheap regex scan + path resolution — cheaper than the validation pass `check` already runs. A whole second Lance dataset (manifest, fragments, its own incremental-write story to keep in lockstep with `index.lance`) is a sledgehammer for a list of `(src, dst, type)` triples, and the headline "compose graph + search in one query" never survived the split (it was always BFS → `WHERE file_id IN (...)`, two steps glued).

Instead: store the outbound links extracted from each chunk as a **column on that chunk's row in `index.lance`**.

- Proposed shape: `links: List<Struct{ raw: Utf8, target_file_id: Utf8?, anchor: Utf8? }>`. Resolved link → `target_file_id` is `Some`; broken → `None`, keep `raw` so `info` can report it. This folds the would-be `broken` / `ambiguous` / `anchor_text` / `heading_anchor` / `source_chunk_id` columns into one nested column. Lance/Arrow handles `List<Struct>` (same machinery as the existing nested `data` frontmatter struct).
- **Per-chunk granularity is correct**, not a compromise: links physically live in a chunk, so storing them there is faithful and non-redundant (a file-level column would repeat the list across every chunk of the file). File→file edges (the graph nodes are files — heading anchors are an attribute, not a separate node) come from a **group-by `file_id`** during the petgraph projection — part of the O(corpus) projection that already feeds petgraph, so it costs nothing extra.
- **Frontmatter `is_reference` links are nearly free**: those fields are already stored per-file in the `data` struct. Don't duplicate them into the `links` column — the projection parses the wikilink out of `data.<field>` on read. Only *body* links need the new column.
- Rides TODO-0173's incremental writes for free: the column is part of the chunk row, so rewriting a file's chunks rewrites its links in the same operation — consistency is automatic, no second incremental path.

**The one tradeoff, stated honestly:** a column on `index.lance` only exists after `mdvs build`, so the *persisted, queryable* graph couples to the search / build layer. The two-layer invariant is preserved by splitting responsibilities: `check` computes broken / ambiguous links **on the fly** during its existing scan (model-free reference integrity, no build required → validation stands alone); `mdvs build` persists the `links` column; the query axes read it.

## Traversal — petgraph at query time

Project `(file_id, links)` + group-by + build `petgraph::DiGraph` per query. Estimated: ~10ms at 1K files, ~100ms at 10K, ~1s at 100K, ~10s at 1M. At the corpora mdvs actually runs (Refractions 549, example_kb 45, K8s 1,669) this is sub-second and moot. The only thing that would bite is composing a graph filter into *every* search (rebuild petgraph per query) — that's the measured signal to add a deferred disposable cache (content-hash-keyed, rebuilt from the column). Same deferral discipline mdvs uses elsewhere: on-the-fly is the v0 answer; the cache stays unbuilt until a real corpus crosses ~100K files.

## UX — folds into `search` / `check` / `info` (no `mdvs graph` command)

The graph is "just another part of the data," so it doesn't get its own verb. A **search is a set of composable narrowing operators over the document set**; each operator is optional, any subset composes. Three axes, each using the idiom that fits it:

| Axis | What it does | Surface |
|---|---|---|
| Relevance | rank by similarity to an anchor | query string (to text) · `--related-to <file>` (to a doc, model-free via stored embedding) |
| Attributes | filter by typed frontmatter | `--where "<SQL>"` (existing) |
| Relationships | filter by graph connection | `--connected-to <file> --depth N` · `--backlinks-of <file>` |

```
mdvs search                                       # all docs
mdvs search --where "status = 'active'"           # attributes only
mdvs search "calibration drift"                   # relevance only
mdvs search --backlinks-of protocols/spec.md      # relationships only
mdvs search "calibration drift" \                 # all three composed
  --where "status = 'active'" \
  --connected-to protocols/spec.md --depth 2 --mode hybrid
```

The would-be `mdvs graph` namespace dissolves entirely:

- **Navigation / filtering** → flags on `search` (above). A REPL "walk" mode is out — it would violate the "no interactive prompts until 1.0" invariant. Human spatial navigation belongs to a future TUI / GUI, not the CLI; the CLI stays one-shot + `--output json`, which *is* the right shape for the agent audience.
- **Integrity** (broken, ambiguous) → these are *violations*, so they belong in `check`, model-free. Never a search.
- **Stats** (edge counts, most-linked) → `info`.

## What to extract

Three link forms, all resolved to file paths within the scanned root; external `http(s)://` links are ignored.

1. **Markdown links**: `[text](relative/path.md)` — standard GitHub / mdBook style. Relation `markdown_link`.
2. **Wikilinks**: `[[page]]`, `[[page|display text]]`, `[[page#heading]]` — Obsidian / wiki style. Relation `wikilink`; display text → `anchor`, heading → a heading anchor (both carried in the `links` struct).
3. **Frontmatter references**: any frontmatter field whose schema declares `is_reference = true` produces `fm_reference` edges tagged with the field name — derived from the stored `data` struct, not re-scanned.

Resolution rules (configurable under `[graph]`):

- **Wikilink resolution.** Default strict: ambiguous wikilinks become `LinkAmbiguous` violations in `check` with one resolution applied (lexicographic-first by default). `resolve = "proximity"` opts into Obsidian-style closest-match resolution.
- **Broken links**: surface as `LinkBroken` in `check`. Soft by default (warning); `strict = true` makes them violations. The unresolved `raw` target is retained in the `links` column so `info` can list broken links.

## Frontmatter reference declaration

New attribute on `[[fields.field]]` entries:

```toml
[[fields.field]]
name = "parent"
type = "String"
is_reference = true              # parse the value as a wikilink → fm_reference edge

[[fields.field]]
name = "related"
type = "Array(String)"
is_reference = true              # array form: each element parsed as a wikilink
```

When `is_reference = true`, the field's string value(s) are parsed as wikilinks during the existing frontmatter validation pass — no separate scan, and no separate storage (the edges derive from the `data` struct on read).

## Config

New `[graph]` section in `mdvs.toml`:

```toml
[graph]
enabled = false                  # whole graph layer; default off in v0 to ease rollout
resolve  = "strict"              # "strict" | "proximity"
strict   = false                 # broken links become violations when true (otherwise warnings)
```

## Semantic links on the fly

"Related notes" is **k-NN, not clustering** (clustering partitions all files into unstable, single-label communities; k-NN gives a per-file ranked neighbour list — what we want). It's literally the vector search `mdvs search` already runs, using a file's *own stored embedding* as the query vector:

1. Fetch file X's chunk embeddings — already in `index.lance`.
2. Run the existing `nearest_to` against the index, filter same-file, aggregate to file-level, take top-K.
3. One vector search — the cost of a single search — and **model-free for an existing file** (read its stored embedding; never load the embedder).

So [TODO-0171](TODO-0171.md)'s premise — precompute every file's top-K and *store* it — is a premature optimization for a query we can already answer on demand. It **folds into `search --related-to`** and retires its stored-table design. Materializing all-pairs top-K only pays off for whole-graph views (e.g. an overview that draws semantic edges for every file = N searches → make it opt-in / compute-at-export-and-discard). If clusters are ever wanted, they're a throwaway *render-time visual aid* only, never stored.

## Prior art — filter vs rank is the real organizing principle

Surveyed how production systems put relevance + attribute + relationship in one query. Two philosophies: **(A) one query representation** with operators for all three — Cypher/GQL (`SEARCH ... IN VECTOR INDEX ... WHERE`, graph-first), SurrealQL (SQL + graph arrows + vector/FT, SQL-first), SQL/PGQ `GRAPH_TABLE(... MATCH ...)` (SQL:2023, graph as a sub-clause), Weaviate GraphQL (orthogonal clauses + cross-references), Vespa YQL (`nearestNeighbor()` + boolean + `rank()`); **(B) multiple retrievers + fusion** — GraphRAG merges separate vector/graph/keyword retrievers with Reciprocal Rank Fusion (`Σ 1/(k+rank_i)`, rank-based so scores needn't share a scale).

The cross-cutting lesson, more important than syntax: **each layer is either a hard filter (narrows, boolean) or a soft ranker (orders the survivors)**, and the clean systems make that explicit (Vespa's `rank()` = "retrieve by A, score by B"; Neo4j's pre/in/post-filter taxonomy). Attributes = filter; relevance = ranker; relationship = *either* (filter: "≤N hops"; rank: link-proximity / PageRank boost). **mdvs already embodies this** — `--where` is a filter, query+`--mode` is a ranker, `hybrid` is already RRF fusion of two rankers. So graph-as-filter via flags is the natural v0; graph-as-*boost* folds into the existing hybrid/RRF machinery as a later step.

Sources: [Neo4j vector+filter](https://neo4j.com/blog/genai/vector-search-with-filters-in-neo4j-v2026-01-preview/), [SurrealDB KG-RAG](https://surrealdb.com/blog/knowledge-graph-rag-two-query-patterns-for-smarter-ai-agents), [DuckPGQ SQL/PGQ](https://duckpgq.org/documentation/sql_pgq/), [Weaviate hybrid](https://docs.weaviate.io/weaviate/search/hybrid), [Vespa NN search](https://docs.vespa.ai/en/querying/nearest-neighbor-search), [RAG patterns 2026 / RRF](https://ailearningguides.com/rag-production-patterns-2026/).

## Comparative landscape — graphify, and why mdvs diverges

mdvs's graph design is fixed mostly by mdvs's own invariants — Lance single-stack storage, the validation / search two-layer split, offline deterministic indexing, `file_id` identity, single-source-of-truth with a recreatable `.mdvs/`. Those already settle most of the decisions. It's still worth situating the design against the closest neighbouring tool, [graphify](https://github.com/safishamsi/graphify) (a knowledge-graph builder for agent-curated corpora), because the comparison is clarifying rather than derivative: on every axis that matters, mdvs lands somewhere *different*, and for reasons intrinsic to mdvs.

**Persistence — a different problem, so a different answer.** graphify keeps an in-memory NetworkX graph and persists it to a single `graph.json` (node-link JSON) plus derived `graph.html` / `graph.svg` / report / Obsidian vault; it stays cheap via a content-hash extraction cache. Crucially, that `graph.json` *is graphify's database* — it has no other store, because its nodes are extracted concepts with nowhere else to live. mdvs is the opposite situation: the graph derives from data mdvs already holds (the markdown corpus + `index.lance`), so a standalone graph file would be a redundant second store to keep in lockstep, and JSON is worse than Lance for it (parse-whole-file per query, no pushdown, no incremental writes). mdvs's storage therefore stays a `links` column on the chunk rows it already maintains; a `graph.json` / `graph.html` appears only as a *derived export* (`mdvs export`), which as a bonus gives Obsidian / tooling interop. Two robustness points stand on their own as mdvs requirements — and it's corroboration, not borrowing, that a mature neighbour independently reached the same: derived graph state should refuse to silently shrink (a dropped file mustn't wipe half the graph), and directional edges must survive serialization (use a directed representation — `petgraph::DiGraph` — rather than reconstructing direction afterwards).

**Edge-building — explicit AND implicit, but implicit is an LLM judgment, not embeddings.** graphify builds from two sources: *structural / explicit* (deterministic tree-sitter AST — `calls / imports / contains / …`) and *semantic / implicit* (LLM subagents emit `semantically_similar_to` etc. for prose). "No vector embeddings" is literally true — similarity is an **LLM decision**. The load-bearing contrast for mdvs, where each tool uses the machinery it already has for the implicit layer:

| | graphify | mdvs (planned) |
|---|---|---|
| Implicit-edge mechanism | LLM subagent judgment | cosine k-NN over stored embeddings |
| Cost | one LLM call per chunk | one vector search, model-free at query time |
| Output | *named, typed, scored* relations | *unlabeled* similarity + cosine score |
| Refresh | expensive → gated re-extraction | cheap → recompute on demand |
| Node identity | derived string id; must match the AST id or ghost-dupes | stable content-addressed `file_id` |

Implications: (a) mdvs's **file-level node granularity** (nodes are files) is *validated* — graphify's biggest correctness headache is concept-node identity across two extractors, which mdvs avoids because nodes are files. (b) mdvs should **not** put an LLM in its own index path (breaks offline / standalone-validation); k-NN over the embeddings mdvs already stores is the native, lighter implicit layer. (c) Because mdvs's *callers* are agents, the graphify-style "name the relationship" judgment can live in the calling agent — mdvs supplies the edges, the agent supplies the naming on demand.

## Online search — a separate source axis, deferred

"Search online" is a fourth dimension — not another operator but another *source* (`--source local|web|both`). It changes mdvs's identity (breaks offline / single-binary / no-external-services; needs API keys; web results aren't markdown-with-typed-frontmatter, so `--where` / schema / validation don't apply). Treat as explicitly opt-in, decided as its own thing later; it must not shape the local SQL + semantic + graph unification.

## Still open (do not implement until settled)

1. **Representation for v0**: orthogonal flags (Weaviate-style, zero parser work, no cross-layer `OR`) vs graph predicates embedded in the `--where` SQL language (SurrealQL / SQL/PGQ-style, full cross-layer boolean, grows the `--where` translator). Leaning flags for v0, SQL-embedded as a later step.
2. **Filter vs rank per graph axis**, and whether graph-as-boost (PageRank / link-proximity fused via RRF) is in v0 or deferred.
3. **Composition order** for `--connected-to` + a relevance query (pre- / in- / post-filter, per Neo4j). Spec it; don't leave implicit.
4. **`--related-to` presentation** when merged with explicit links: one ranked list vs bucketed ("Explicit" / "Semantic neighbours").
5. **Exact `links` column struct** — keep `anchor` / heading? `source_line`?
6. **Does semantic `--related-to` ship in v0** or land after the explicit-link axes.
7. **GUI** (separate, not-yet-written TODO): overview-poster (self-contained static HTML, force-directed whole-graph view) vs navigation-cockpit (ratatui TUI — strong at link-walking, weak at drawing graphs); local `mdvs serve` held as a later option (daemon model fights invoke-and-exit). What it visualizes first (link graph needs this TODO; a faceted browse / validate view works on today's data). mdvs should offer `mdvs export --format graphjson|html` as a *derived* artifact from the `links` column.

## Scope (v0) — what ships

- Extract markdown links, wikilinks, and `is_reference` frontmatter references during `mdvs build`; persist as the `links` column.
- `check`: `LinkBroken` / `LinkAmbiguous` violations, computed on the fly (model-free, no build required).
- `search`: `--connected-to <file> [--depth N]`, `--backlinks-of <file>`, `--related-to <file>`, composed with the existing modes and `--where`.
- `info`: edge count, broken / ambiguous counts, top-N most-linked files.
- `[graph]` config (`enabled` default false, `resolve`, `strict`) and the `is_reference` schema attribute.
- Multi-hop via petgraph projection, no cache.

Deferred, gated on measured evidence: a content-hash-keyed disposable traversal cache (only if a real corpus crosses ~100K files); PageRank as an optional ranking boost (only if semantic ranking alone leaves quality on the table). **`--connected-to --depth > 1` needs a benchmark** on a 50K+ synthetic corpus before promotion — `WHERE file_id IN (...)` with thousands of literals may not push down cleanly.
