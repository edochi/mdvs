# TODO-0080: Shared step output structs

## Summary

Define the output structs for each processing step. These are shared across commands — e.g., `ScanOutput` is used by check, build, init, and update.

## Details

### Step output structs

| Struct | Used by | Key fields |
|--------|---------|------------|
| `ReadConfigOutput` | check, build, search, info, update, clean | config path |
| `ScanOutput` | check, build, init, update | files found, glob |
| `ValidateOutput` | check, build | files checked, violations, new fields |
| `InferOutput` | init, update | fields inferred, fields changed |
| `ClassifyOutput` | build | new/edited/unchanged/removed counts |
| `LoadModelOutput` | build, search | model name, revision |
| `EmbedFilesOutput` | build | files embedded, chunks produced |
| `EmbedQueryOutput` | search | query text |
| `ExecuteSearchOutput` | search | results count |
| `WriteConfigOutput` | init, update | config path |
| `WriteIndexOutput` | build | files written, chunks written |
| `ReadIndexOutput` | build, search, info | metadata, file/chunk counts |
| `DeleteIndexOutput` | clean | files removed, bytes freed |

### Design notes

- Each struct implements `Serialize` for JSON output
- Each struct needs a `format_text(&self, verbose: bool) -> String` method (or implements a trait) for text rendering as a step line
- Some structs are very simple (ReadConfigOutput might just be a path) — that's fine, uniformity matters more than minimalism
- The existing `CheckResult`, `BuildResult`, etc. become the *command result* — separate from step outputs

## Files

- `src/pipeline.rs` (or `src/output.rs`) — all shared step output structs
