# TODO-0170: Incremental check/update via `.mdvs/cache.toml`

## Problem

`mdvs search` with the default config runs `auto_update` + `auto_build`
before every query. On large corpora this is catastrophic: measured
during TODO-0166 benchmarking on the full kubernetes/website docs
(1,669 files), **default-config search takes ~21 seconds per query**
(machine-load-inflated; still seconds on a quiet machine — unusable for
interactive search). The same query with `--no-update --no-build` is
**~570 ms**.

| corpus | files | mdvs default search | mdvs engine-only search |
|---|---|---|---|
| example_kb | 45 | ~110 ms overhead | ~210 ms |
| K8s docs | 1,669 | **~21 s** | ~570 ms |

The overhead scales linearly with corpus size, making the default
configuration unusable on real-world-sized vaults.

## Design goal — keep all auto-steps on by default

Target audience is **general users** (Obsidian-style), not just
technical users or CI. The experience must be seamless: drop a note,
run a search, it just works — no remembering to `update` then `build`
then `search`. So `auto_update` + `auto_build` stay **on by default**.
The job here is to make them cheap enough that "on by default" doesn't
cost seconds.

## Root cause (verified in code)

The search path does **not** call the `update` or `check` commands. It
calls `build::build_core(..., auto_update, ...)` (`search.rs:96`), which
(`build.rs:282`):

1. scans — reads every file
2. `if auto_update`: `InferredSchema::infer(&scanned)` — inference
   (`build.rs:314`)
3. `check::validate(&scanned, config, false)` over **all** files
   (`build.rs:387`) — validation
4. embed — already incremental via `content_hash`

So the cost is: read every file + infer over all + validate all. Steps
1–3 run full on every search. (`update` the *command* is pure inference
and does not validate; `check` the *command* is pure validation. The
search path goes through `build_core`, which does both internally.)

## Correctness principle

A file's validation result is a **pure function of (its content, the
schema)**. If both are unchanged, the result is provably identical — so
it can be cached and trusted. Likewise a file's inference contribution
is a pure function of its content. Cache keyed on `content_hash` +
`schema_hash`; detect "unchanged" cheaply with `(mtime, size)` (a
`stat`, no file read), confirm with `content_hash` when stat differs.

Per-file cost for an **unchanged** file drops from
`STAT + READ + PARSE + VALIDATE` to just `STAT` (~1000× cheaper). On a
typical repeated search nothing changed, so the read/parse/validate set
is empty.

## The artifact — `.mdvs/cache.toml`

A gitignored cache (not a "lock" — it exists for speed, not committed
reproducibility), living in `.mdvs/` alongside `index.lance/` but
**independent** of it. Model-free: `check`/`update` use it without the
embedding index or model. In CI (fresh checkout, no `.mdvs/`) it's
simply absent → full validation, preserving the validation/search
decoupling.

### Contents (TOML, Cargo.lock-style)

The whole cache is keyed to a single `schema_hash` — if it matches the
current `mdvs.toml`, every entry is trusted; if not, re-validate all and
rewrite. So entries need no per-file `validated` flag.

```toml
# .mdvs/cache.toml — generated, do not edit. Safe to delete (rebuilt next run).
version = 1

[meta]
mdvs_version = "0.6.2"
schema_hash = "xxh3:a1b2c3"          # the mdvs.toml schema these entries validated against
generated_at = "2026-05-28T10:30:00Z"

[[files]]
path = "concepts/overview.md"
mtime = { secs = 1716800000, nanos = 123456789 }   # full precision, exact-match compare
size = 4096
content_hash = "xxh3:deadbeef"
# inference contribution (the update payload): fields present + observed shape
fields = [
  { name = "title",  type = "String",  nullable = false },
  { name = "weight", type = "Integer", nullable = false },
]
```

Deliberate exclusions, to keep it small:
- **No field values.** Categorical/range/`reinfer` need values, but
  `reinfer` does a targeted full read. `fields` carries only
  name/type/nullable — all the default add-only `update` needs.
- **No `allowed`/`required` globs.** Derived by aggregating which paths
  have each field across entries; the path is the entry key.

Format: TOML for v1 (debuggable). At very large scale (100k+ files) the
TOML gets chunky; since the file is gitignored and never hand-edited,
swapping to JSON or a compact binary is a cheap future change — keep the
serializer behind one module.

## The add-only insight (makes incremental `update` tractable)

Default `update` is **add-only** (verified `cmd/update.rs:197-309`): it
preserves every existing field verbatim — including hand-tuned
constraints — and only appends genuinely new fields. Never modifies,
narrows, or removes. Only explicit `update reinfer [field]` touches
existing fields.

So incremental `update` needs **no subtraction machinery**. The cache's
role is *memory of unchanged files' field membership*: incremental
update = `(unchanged files' cached field sets) ∪ (changed/new files'
fresh field sets)`, then "any field not in mdvs.toml and not ignored →
add." This also fixes the one case naive stat-skipping would miss: a
field living only in *unchanged* files but absent from mdvs.toml (e.g.
hand-deleted) — the cache remembers it without a re-read.

## How each command relates to the cache

- **`check` (standalone) — cache-free.** Neither reads nor writes the
  cache; always full-validates. Correct, model-free, CI-friendly. A
  command shouldn't write an artifact it doesn't consume. The standalone
  validation path is *not* incremental — fine, because CI runs fresh and
  the search path never calls standalone `check`.
- **`update` — cache owner (inference payload).** Stat-walk; reuse
  cached field membership for unchanged files, re-read changed/new,
  merge (add-only). Builds the cache from a full scan when absent.
- **`build` — cache owner (metadata/validation) + runs `update`.** Uses
  the cache to read+validate only changed files (same `validate()` core
  as `check`, `check.rs:302`, but cache-driven file selection), trusting
  unchanged entries. Embedding stays incremental via `content_hash`.
- **`init` — primes the cache** from its scan (it already has the data),
  unless `--no-cache`.

The search path stays fast because `build_core`'s validation is
cache-aware — not because standalone `check` is. Same core, two callers,
different file selection.

Steady-state "nothing changed" search drops from O(read+validate of N)
to O(stat of N): ~21 s → tens of ms, correct on in-place edits
(mtime/size + content_hash), new fields in edited files (edit changes
stat → re-read → field surfaces), and schema changes (schema_hash
mismatch → re-validate all).

## Opt-out: `[scan].validation_cache`

- **`[scan].validation_cache`** config field — `true` by default. When
  `false`, no command reads or writes the cache (always full scan).
- **`init --no-cache`** is the *only* flag — it writes
  `validation_cache = false` into the generated `mdvs.toml` so the
  opt-out persists across all subsequent commands. No other command
  carries a cache flag.
- **Graceful degradation regardless of the setting:** if the cache is
  enabled but unwritable (read-only FS), warn and continue with a full
  scan — never crash. This is the load-bearing safety net; the explicit
  opt-out is a secondary convenience (CI cleanliness, read-only,
  determinism/debugging).

(Name note: `validation_cache` is slightly narrow since the cache also
accelerates `update`'s inference — flagged in case a broader name is
preferred.)

## Why not a folder-level Merkle tree

Hashing folders to skip unchanged subtrees is the right instinct but
hits a filesystem wall: **directory mtimes don't change on in-place file
edits** (APFS/ext4 bump a dir's mtime only on add/remove/rename). So a
folder hash from directory mtimes would miss content edits — fatal for
validation; and a folder hash from file stats saves nothing, because you
stat every file to build it. The per-file stat walk is already cheap
(~ms for thousands of files); the expensive thing we eliminate is
read+parse+validate, which the per-file cache handles directly. A
directory-mtime tree could skip stat-ing subtrees only at 100k+ scale
(accepting a periodic full rescan for in-place edits) — future
optimization, not now.

## Design questions still open (for implementation time)

- **Merge equivalence test.** Incremental `update`/`build` must produce
  results byte-identical to the `--force` full path, or they drift.
  Needs a property test: random edits → incremental == full.
- **`mtime` precision/portability.** Store full precision (`secs` +
  `nanos`); `git checkout` rewrites mtimes → harmless false-positives
  (extra re-validation, never a miss); `content_hash` is the tiebreaker
  when stat differs.
- **`clean` wipes the cache** (it deletes `.mdvs/`). Consistent with
  "clean resets derived state," but the next `update`/`build` is then a
  cold full scan. Confirm acceptable; document.
- **`validation_cache` name** — keep, or broaden (it serves update too)?

## Out of scope

- Filesystem watchers (daemon required; mdvs is one-shot).
- The folder Merkle (future 100k+ scale only).
- Turning auto-update/auto-build **off** by default — deliberately
  rejected; the target is seamless general-user UX. This makes "on by
  default" cheap instead of removing it.
- Subtraction/removal machinery for inference — default `update` is
  add-only; only `reinfer` touches existing fields and it full-reads its
  named fields.

## Impact

High priority and pre-promotion. A first-time user pointing mdvs at a
big docs tree and searching would see multi-second latencies and
reasonably conclude the tool is slow — when the engine is sub-second.
Discovered during TODO-0166 benchmarking.
