# TODO-0149: JSON Schema as canonical schema + tomljson crate + preprocessor pipeline

## Summary

Adopt JSON Schema 2020-12 as the canonical internal representation for field validation. Split into three internal waves:

- **Wave A вҖ” `tomljson` crate**: new workspace member providing **lossless bidirectional translation between TOML and JSON-shaped data**, with explicit handling of TOML's impedance mismatches (no native null, no top-level non-table, signed-64-bit integer cap). Designed with JSON Schema 2020-12 as the motivating use case вҖ” anyone with arbitrary JSON data can use the same translator. No validation, no opinions, no JSON Schema awareness in the implementation.
- **Wave B вҖ” mdvs adoption**: mdvs uses JSON Schema internally for validation via the `jsonschema` crate. `mdvs.toml` becomes a user-friendly DSL that translates to canonical JSON Schema. Preprocessor pipeline handles leniency. Subsumes TODO-0008, TODO-0010, TODO-0145.
- **Wave C вҖ” type naming + object flattening alignment**: align `mdvs.toml` type names to JSON Schema (`string`, `integer`, `number`, `boolean`, `array`, `object`) and flatten nested objects per TODO-0097.

## Design decisions (settled)

### Two-layer representation

| Layer | Form | Role |
|---|---|---|
| `tomljson` (canonical) | **Nested** TOML mirroring JSON Schema 1:1 вҖ” `[fields.field.properties.baseline.properties.wavelength]` | Pure translator. Full JSON Schema expressiveness. Reusable crate. |
| `mdvs.toml` (DSL) | **Flattened** `[[fields.field]]` with dotted names вҖ” `name = "calibration.baseline.wavelength"` (per TODO-0097) | User-friendly config surface. Translates to canonical JSON Schema internally. |

The flattened DSL is mdvs-specific UX. The canonical form is what `jsonschema` validates against.

### Type naming

Aligned to JSON Schema: `string`, `integer`, `number`, `boolean`, `array`, `object`, `null`. Replaces the current `String`, `Integer`, `Float`, `Boolean`, `Array(...)`, `Object({...})`. Wave C is the breaking config change.

### What `jsonschema` handles (delegated)

Everything per-value:

| Concern | JSON Schema keyword(s) |
|---|---|
| Type checking | `type` |
| Nullable | `"type": ["string", "null"]` (type union) |
| Categories | `enum` |
| Numeric bounds | `minimum`, `maximum` |
| String length | `minLength`, `maxLength` |
| Array length | `minItems`, `maxItems` |
| Regex pattern | `pattern` (ECMA-262 via `fancy-regex`) |
| Array element schema | `items` |
| Object structure | `properties` (recursive) |

### What mdvs keeps (not delegated)

| Concern | Why |
|---|---|
| `validate_for_type` | Reject misapplied constraints at config load. JSON Schema silently ignores inapplicable keywords; mdvs wants a hard error. |
| `conflicts_with` | Reject contradictory constraints (e.g., `min > max`). |
| Path-aware rules | `allowed`/`required` globs are per-file-path, not per-value. Not expressible in JSON Schema. Encoded as `x-mdvs-allowed` / `x-mdvs-required` extensions when exporting. |
| Preprocessor pipeline | Value coercion/normalization before validation. mdvs domain logic. |
| Error mapping | Translate `jsonschema` errors to mdvs `ViolationKind` with human-readable messages. |

### Preprocessor pipeline (three-stage)

Stages are independent: each has its own trait, its own list of registered preprocessors, its own ordering. A user can register handlers in any combination.

#### Stage 1: Field-name preprocessors

Operate on keys only. Trait:

```rust
pub trait FieldNamePreprocessor: Send + Sync {
    fn name(&self) -> &str;
    fn process(&self, key: &str) -> Cow<'_, str>;
}
```

Built-in candidates (none in v0; all future):
- Lowercase / uppercase
- Trim whitespace
- Replace spaces / dashes with underscores

Practical use case: a user has frontmatter `Title: "Hello"` but their schema declares `title: { type: "string" }`. A lowercase preprocessor closes the gap.

#### Stage 2: Per-field value preprocessors (initial scope)

Operate on one (key, value) pair at a time. Trait:

```rust
pub trait FieldValuePreprocessor: Send + Sync {
    fn name(&self) -> &str;
    fn process(&self, value: &Value, field_schema: &Value) -> Result<Value>;
}
```

The preprocessor sees the field's individual JSON Schema sub-tree (extracted from the full schema by the pipeline runner using the field name as a JSON pointer).

**Built-in (lenient mode, default):**

| Preprocessor | What it does |
|---|---|
| `coerce_to_string` | Serialize non-string values to JSON strings for string fields (String-is-top-type) |
| `widen_int_to_float` | Accept integers in number fields |

**Future built-ins (candidates):**

| Preprocessor | What it does |
|---|---|
| Lowercase / uppercase | Normalize string case before enum / pattern matching |
| Trim whitespace | Strip leading/trailing whitespace |
| Date normalization | Parse various date formats into canonical RFC 3339 |

#### Stage 3: Per-document preprocessors (future, design-only for now)

Operate on the whole frontmatter object, with full-schema visibility. Trait:

```rust
pub trait DocumentPreprocessor: Send + Sync {
    fn name(&self) -> &str;
    fn process(
        &self,
        doc: &serde_json::Map<String, Value>,
        schema: &Value,
    ) -> Result<serde_json::Map<String, Value>>;
}
```

Use cases this stage exists to enable:

- **Cross-field coherence**: "if `status: published` then `published_at` must be set" вҖ” a preprocessor could fill in `published_at = today()` when missing.
- **Computed defaults**: "if `slug` is missing, derive from `title`."
- **Field interdependencies**: "if `archived: true`, normalize `status: archived` regardless of what's there."

Stage 3 is not implemented in v0 of Wave B but the trait is defined and the pipeline runner calls it (with an empty preprocessor list initially). This means future additions don't require a pipeline rearchitecture.

#### Mode handling

```rust
pub enum Mode {
    Lenient,  // run all stages (default)
    Strict,   // skip all stages вҖ” raw value goes directly to jsonschema
}
```

`--strict` mode (TODO-0150): pipeline is bypassed entirely. The user's frontmatter must already match the schema exactly.

#### Lua custom preprocessors (TODO-0144)

User-defined Lua functions plug into any of the three stages. Each stage gets a configurable list of Lua scripts in addition to the built-ins. Same trait surface вҖ” Lua adapter implements `FieldValuePreprocessor` etc. by invoking a registered Lua function.

#### Pipeline runner

```rust
pub struct Pipeline {
    name_preprocessors: Vec<Box<dyn FieldNamePreprocessor>>,
    value_preprocessors: Vec<Box<dyn FieldValuePreprocessor>>,
    document_preprocessors: Vec<Box<dyn DocumentPreprocessor>>,
}

impl Pipeline {
    pub fn apply(
        &self,
        raw: serde_json::Map<String, Value>,
        schema: &Value,
        mode: Mode,
    ) -> Result<serde_json::Map<String, Value>> {
        if matches!(mode, Mode::Strict) { return Ok(raw); }

        // Stage 1: rewrite keys
        let renamed: serde_json::Map<String, Value> = raw.into_iter()
            .map(|(k, v)| (self.apply_name_stage(&k).into_owned(), v))
            .collect();

        // Stage 2: rewrite each value with its field schema
        let mut processed = serde_json::Map::with_capacity(renamed.len());
        for (k, v) in renamed {
            let field_schema = field_schema_for(schema, &k);
            let new_v = self.apply_value_stage(&v, field_schema)?;
            processed.insert(k, new_v);
        }

        // Stage 3: cross-field
        let final_doc = self.apply_document_stage(processed, schema)?;
        Ok(final_doc)
    }
}
```

The runner is the only place that knows about all three stages. Adding a fourth stage in the future is mechanical.

### Logical pipeline per command

#### `check [path]`

```
1. Load mdvs.toml вҶ’ MdvsToml DSL
2. Translate DSL вҶ’ canonical JSON Schema вҶ’ compile jsonschema::Validator
3. Scan files вҶ’ ScannedFiles (YAMLвҶ’JSON happens here, errors surfaced)
4. Per file:
   a. Run preprocessor Pipeline (stages 1-3) on the frontmatter
   b. Validator.validate(processed_frontmatter)
   c. Map jsonschema::ValidationError вҶ’ ViolationKind
5. Apply path-aware rules (allowed/required globs) вҖ” mdvs-side
6. Aggregate violations вҶ’ CommandOutput
```

#### `init [path]`

```
1. Scan files
2. Infer:
   a. FieldType widening + distinct-value collection (existing)
   b. Constraint inference: categories, ranges, lengths, patterns
   c. Build canonical JSON Schema from inferred types  вҶҗ uses schema/json_schema.rs
   d. Translate canonical вҶ’ MdvsToml DSL
3. Write mdvs.toml
4. (optional) build
```

With `--from-schema <path>`:

```
1. Parse input schema:
   - JSON file вҶ’ serde_json::from_str
   - canonical TOML file вҶ’ tomljson::from_str
2. Translate canonical вҶ’ MdvsToml DSL
3. Write mdvs.toml (+ optional build)
```

#### `update [path]`

Unchanged shape; inference logic gains length/pattern support (constraint inference module).

#### `build [path]`

Calls `check` internally (existing). The check call now goes through the new pipeline.

#### `search`, `info`, `clean`

Untouched.

#### `export-schema [path] [--output-file FILE] [--format json|toml]`

```
1. Load mdvs.toml вҶ’ MdvsToml DSL
2. Translate DSL вҶ’ canonical JSON Schema (serde_json::Value)
3. Emit:
   - --format=json вҶ’ serde_json::to_string_pretty (default)
   - --format=toml вҶ’ tomljson::to_string
```

### YAML вҶ’ JSON conversion (frontmatter вҶ’ validatable value)

mdvs reads YAML frontmatter from `.md` files and validates the resulting *JSON-shaped* value against a JSON Schema. JSON Schema validators (including the `jsonschema` crate) operate on `serde_json::Value`, not on YAML. The YAMLвҶ’JSON conversion is therefore an unavoidable step before validation can happen.

**Asymmetry with the TOML side**: TOML and JSON have a structural impedance mismatch (TOML lacks null, top-level non-table values, and integers > i64::MAX) that requires a dedicated bridging crate (`tomljson`). YAML and JSON are much closer вҖ” JSON 1.0 is a strict subset of YAML 1.2 вҖ” so `serde_yaml`'s built-in `Pod::deserialize::<serde_json::Value>()` handles ~95% of the conversion. There is no need for a `yamljson` companion crate.

**Where the YAMLвҶ’JSON gap leaks in mdvs today**

The current code at `src/discover/scan.rs:126` swallows conversion failures silently:

```rust
let data = parsed.data.and_then(|d: Pod| {
    let json: Value = d.deserialize().ok()?;        // вҶҗ ".ok()?" drops the entire frontmatter
    if json.is_object() { Some(json) } else { None }
});
```

If any single field has a value that can't fit in `serde_json::Value` (the only realistic cases: `.inf` / `-.inf` / `.nan` floats, or non-string mapping keys), `Pod::deserialize` returns an error, `.ok()?` discards it, and the **entire frontmatter** is treated as missing. Other valid fields in the same file are also lost.

**Required Wave B work**

Replace the silent drop with explicit handling:

| YAML feature | Wave B handling |
|---|---|
| `.inf` / `-.inf` / `.nan` float | Error with `ViolationKind::FrontmatterUnrepresentable { field, reason }` indicating which field and why. Same policy as `tomljson`'s decode side. |
| Non-string mapping keys | Error with same kind; very rare in real frontmatter. |
| Anchors / aliases | Resolved by `serde_yaml` at parse time вҶ’ already JSON-compatible. No action needed. |
| Tags (`!!str`, custom) | Flattened by `serde_yaml` to base types. No action needed. |
| Implicit dates (`published: 2026-05-04`) | Kept as strings in YAML 1.2 mode (`serde_yaml` default). No action needed unless we want to coerce to a normalized form (preprocessor candidate). |
| Comments | Discarded at parse time. Not data, no action. |
| Multi-document `---` | Not applicable to single-document frontmatter. |

The fix is two parts:

1. **In `scan.rs`**: surface the deserialize error instead of `.ok()?`. Identify which field caused the failure if possible (may require a custom `Deserialize` walker since `Pod::deserialize` to a generic Value returns an opaque error).
2. **In the validation layer (Wave B)**: define a new `ViolationKind` variant for unrepresentable frontmatter values, distinct from "frontmatter missing" or "value of wrong type."

This is a small but real correctness improvement that the JSON Schema integration enables вҖ” once we adopt `jsonschema`, we're committing to "the value must be representable as JSON," and silent drops contradict that promise.

**Documentation requirement (book/)**: the user guide for the validation feature must explain that frontmatter values are validated against JSON-shaped data, and must list the YAMLвҶ’JSON conversion rules so users understand why `.inf` isn't accepted.

### Import/export

`mdvs.toml` is the source of truth for the CLI; the canonical JSON Schema is the internal representation. Two flows:

- **Export**: `mdvs export-schema [path] [--output-file schema.json]` writes the JSON Schema document mdvs uses internally. Same for `tomljson` export вҖ” produces canonical TOML.
- **Import**: `mdvs init` and `mdvs update` accept `--from-schema <path>` (JSON or canonical TOML via `tomljson`). The schema is translated into `mdvs.toml`'s DSL form. After import the source schema is discarded вҖ” `mdvs.toml` is self-contained.

## Constraint mapping: `mdvs.toml` вҶ” JSON Schema

The translator builds a JSON Schema from `[[fields.field]]` entries. Each field becomes a property in the schema.

### Type mapping (Wave C)

| `mdvs.toml` type | JSON Schema |
|---|---|
| `boolean` | `{"type": "boolean"}` |
| `integer` | `{"type": "integer"}` |
| `number` | `{"type": "number"}` |
| `string` | `{"type": "string"}` |
| `array` + `items` | `{"type": "array", "items": <T-schema>}` |
| `object` (flattened via dotted names) | `{"type": "object", "properties": {...}}` |
| Any + `nullable: true` | `{"type": ["<base-type>", "null"]}` |

### Constraint mapping

| mdvs constraint | Applies to | JSON Schema keyword(s) |
|---|---|---|
| `categories` | string, integer, array of those | `enum` (scalar), `items.enum` (array) |
| `min` / `max` | integer, number, arrays of those | `minimum`/`maximum` (scalar), `items.minimum`/`items.maximum` (array) |
| `min_length` / `max_length` | string, array | `minLength`/`maxLength` (string), `minItems`/`maxItems` (array) |
| `pattern` | string | `pattern` |

### Path-aware rules (extensions)

| mdvs construct | JSON Schema representation |
|---|---|
| `allowed: [glob, ...]` | `x-mdvs-allowed: [glob, ...]` |
| `required: [glob, ...]` | `x-mdvs-required: [glob, ...]` |

Third-party validators silently ignore `x-` extensions.

## `tomljson` crate scope (Wave A)

A standalone crate providing **lossless bidirectional translation between TOML documents and JSON-shaped data** (`serde_json::Value`). Pure translation, no validation, no schema awareness.

### Scope

- **In scope**: any JSON value вҶ” any TOML value, with explicit handling of the impedance mismatches (null, top-level non-table, integer range, datetime asymmetry).
- **Motivating use case**: serializing JSON Schema 2020-12 documents as TOML вҖ” every keyword roundtrips losslessly because the underlying translator is exhaustive.
- **Distinguishes from `jsontoml` / `toml2json`** (existing CLI tools): we are a Rust *library* with bidirectional API and impedance handling, not a one-way command-line filter.

### Out of scope

- Schema validation (delegated to the `jsonschema` crate).
- A schema language for TOML files (Tombi/Taplo's domain вҖ” different problem).
- Any JSON Schema-aware behavior вҖ” the crate doesn't know what `properties` or `oneOf` mean. It treats them as opaque keys.
- Format conversions other than JSONвҶ”TOML (YAML, CBOR, MessagePack вҖ” remarshal's domain).

### How JSON Schema "rides" the translator

JSON Schema 2020-12 is just JSON-shaped data. Every keyword (`oneOf`, `anyOf`, `$ref`, `$defs`, `if`/`then`/`else`, `const`, `enum`, `pattern`, etc.) is a regular JSON value at some path. Because the translator is exhaustive over JSON, it roundtrips every JSON Schema document. The encoding *conventions* below describe how common JSON Schema constructs render in TOML вҖ” but they're emergent, not coded into the crate:

- JSON object keys вҶ’ TOML keys verbatim (`minLength`, `pattern`, `enum`).
- JSON arrays of objects вҶ’ TOML arrays of tables (`[[oneOf]]`).
- JSON keys with `$` or special chars вҶ’ TOML quoted keys (`"$ref" = "..."`, `["$defs".address]`).
- Nested JSON objects вҶ’ nested TOML tables (`[properties.user.properties.email]`).

### Implementation strategy

Use **`toml_writer` (low-level emitter) for serialization** and **`toml::from_str` (parser) + tree walk for deserialization**. The crate is a thin dispatcher around both.

This was settled after surveying the toml-rs crate family:

| Approach considered | Outcome |
|---|---|
| Wrap `toml::Serializer` at the serde trait level | **Infeasible.** `toml`'s internal `MapValueSerializer`, `SerializeDocumentTable`, etc. are `pub(crate)` вҖ” only the outer `Serializer` is wrappable, which doesn't intercept any nested values. |
| Wrap `toml_edit::Serializer` (its internal types are public) | Feasible but ~600 LOC of mechanical trait forwarding, plus a heavier dependency. |
| Pivot through `serde_json::Value` for both directions | Works (~250 LOC) but allocates an intermediate Value tree per call. |
| **Use `toml_writer` directly for serialize, `toml::from_str` + walk for deserialize** | **Chosen.** ~400 LOC, zero allocation on serialize, no state-machine wrapping required. Spike validated in `scripts/test_tomljson_writer.rs`. |

`toml_writer` is the toml-rs project's low-level emitter crate (sibling to `toml`, `toml_edit`, `toml_parser`, `toml_datetime`). It exposes primitives like `open_table_header`, `key`, `keyval_sep`, `value`, `open_array`, `val_sep`, etc., letting us drive TOML emission ourselves while delegating syntactic correctness (string escaping, float formatting, key quoting, multi-line strings) to the upstream crate.

The dispatcher walks a `serde_json::Value` and calls `toml_writer` primitives. When it sees a JSON value TOML can't represent natively (`null`, `u64 > i64::MAX`, top-level non-table), it applies the placeholder/wrapper/error strategy itself.

### API surface (sketch)

```rust
pub struct TomlJsonOptions {
    /// String value used in TOML to mean JSON `null`. Default: "__null__".
    /// Wherever this exact string appears in TOML, the decoder yields JSON null.
    /// The encoder errors if a real string in the input equals this placeholder.
    pub null_placeholder: String,
}

impl Default for TomlJsonOptions { /* null_placeholder = "__null__" */ }

// в”Җв”Җв”Җ Embedded API (caller supplies options) в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
//
// Serialize: walks the value, dispatches to toml_writer primitives.
// Deserialize: parses with toml::from_str, walks Value, applies inverse transforms.

tomljson::to_string_with_opts<T: Serialize>(value: &T, opts: &TomlJsonOptions) -> Result<String>;
tomljson::from_str_with_opts<T: DeserializeOwned>(s: &str, opts: &TomlJsonOptions) -> Result<T>;

// в”Җв”Җв”Җ Standalone API (file = one document) в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
//
// Reads `$tomljson-null` directive from the document if present.
// See "Two-phase parse" below.

tomljson::to_string<T: Serialize>(value: &T) -> Result<String>;
tomljson::from_str<T: DeserializeOwned>(s: &str) -> Result<T>;
```

Because `serde_json::Value` itself implements `Serialize`/`DeserializeOwned`, callers with already-built Values just pass them through:

```rust
let json: serde_json::Value = tomljson::from_str(toml_text)?;
let toml_text: String = tomljson::to_string(&my_json_schema_value)?;
```

The serialize side typically takes a `serde_json::Value` (which implements `Serialize` for free). Internally, the dispatcher pattern-matches on `Value::Null` / `Value::Number` / `Value::String` / etc., so we can intercept impedance gaps directly without going through serde's Serializer abstraction. For typed structs, the user can either pre-convert via `serde_json::to_value`, or we can add a generic blanket implementation later.

### Two-phase parse for the standalone API

TOML's grammar permits top-level keys in any order вҖ” the directive can appear *after* a value that uses the placeholder:

```toml
default = "@@NULL@@"
"$tomljson-null" = "@@NULL@@"   # directive is third in source order
```

A single-pass deserializer cannot know `"@@NULL@@"` is special when it first encounters `default`. The standalone API therefore uses a **two-phase parse**:

1. Parse the input with `toml::from_str` into `toml::Value` (this is what the underlying `toml` crate does anyway вҖ” there is no streaming TOML parser in Rust).
2. Scan the root-level keys for `$tomljson-null`. If present, capture and remove it; if absent, use the default `"__null__"`.
3. Walk the `toml::Value` tree applying the placeholder substitution + root-unwrap, then deserialize the resulting `serde_json::Value` into the user's target type.

The "extra" cost is one Value walk over an already-loaded document. For schema-sized files this is nanoseconds. The whole-file load is forced on us by `toml`'s non-streaming parser, not by our directive feature.

**Serialize side bypasses Value entirely.** With `toml_writer`, we walk the user's `serde_json::Value` directly and emit TOML to a `String`. No `toml::Value` allocation involved.

The crate's `README.md` MUST prominently document:

- Why TOML's data model forces the two-phase parse trade-off (no streaming TOML parser, key-order freedom).
- The full per-direction policy table (encode-side and decode-side handling of every impedance gap).
- Which API entry points are single-pass vs. two-phase.
- What "two-phase" actually costs in time and memory for typical inputs.
- Guidance on when to use which API.
- Why `+inf` / `-inf` / `NaN` error on decode despite storage layers supporting them.
- The intentional asymmetry: TOML grammar has no "key with null value" form, so absent-key and present-with-null must be distinguished by the placeholder convention.

These are first-class design constraints, not footnotes. Users picking up the crate should understand them before integrating.

### Null encoding

TOML has no `null`. The crate represents JSON `null` as a **user-chosen string placeholder**, default `"__null__"`. The placeholder occupies the slot anywhere a null appears вҖ” top-level, nested, or inside arrays вҖ” keeping arrays homogeneous-looking. Examples:

```toml
default = "__null__"
const   = "__null__"
enum    = [1, 2, "__null__"]
```

The decoder (with the placeholder string in hand) walks the parsed TOML and replaces any string equal to the placeholder with JSON `null`.

**Standalone files** may declare a custom placeholder with a `$`-prefixed root directive (matching JSON Schema's meta-key convention `$schema`, `$id`, `$defs`):

```toml
"$tomljson-null" = "@@NULL@@"

type = ["string", "null"]
default = "@@NULL@@"
```

The directive is consumed by the crate and stripped from the JSON output вҖ” JSON Schema validators never see it.

**Embedded use** (mdvs and other library callers) bypasses the directive entirely. The caller passes `TomlJsonOptions` through the API, and the directive does not appear in the surrounding document. For mdvs, the placeholder is a constant the crate uses internally; the directive never appears in `mdvs.toml`.

**Collision behavior** вҖ” error. If a real string value in the input equals the placeholder, the encoder fails with a clear message asking the user to set a different `$tomljson-null` value (or pass a different `null_placeholder` via the API).

### Boolean-schema root encoding

JSON Schema 2020-12 allows the literal `true` or `false` as a complete schema (always-valid / always-invalid). TOML's root must be a table, so the crate wraps non-object roots under a single key `__root__`:

```toml
__root__ = true
```

The decoder unwraps a root table that has exactly one entry named `__root__`. The dunder-style name follows the same convention as `__null__` to make sentinels visually distinct from real keys. Collision behavior matches null: error if a real top-level key is named `__root__`.

### Integer range вҖ” strict spec adherence

TOML integers are 64-bit signed (вҲ’2вҒ¶Ві to 2вҒ¶ВівҲ’1). The TOML 1.1 spec mandates: *"if an integer cannot be represented losslessly, an error must be thrown."* JSON allows arbitrary-precision integers; `serde_json::Number` admits values up to `u64::MAX`. There is exactly one bit of mismatch: the range `(i64::MAX, u64::MAX]`.

The crate adheres to the TOML spec вҖ” values in that range error on encode:

```
input contains integer 18446744073709551615 which exceeds TOML's signed 64-bit range
(i64::MAX = 9223372036854775807); TOML cannot represent this value losslessly
```

We considered (and rejected) two alternatives:
- **Stringified bigint with directive prefix** (mirroring the null placeholder) вҖ” works, but no other TOML tool would understand it; portability lost.
- **Silent f64 fallback** вҖ” corrupts the value above 2вҒөВі; rejected outright.

Real-world JSON data containing values > i64::MAX is rare (snowflake IDs, cryptographic nonces). When they do appear, an explicit error beats silent approximation.

### Per-direction policy table

Bidirectional translation involves asymmetric concerns. The full policy:

#### JSON вҶ’ TOML (encode)

| Concern | Handling |
|---|---|
| `Json::Null` value (anywhere) | Substitute the placeholder string (default `"__null__"`) |
| Top-level non-table value (bool, scalar, array) | Wrap under `__root__` key |
| `serde_json::Number` representable as `u64 > i64::MAX` | **Error** вҖ” TOML's signed 64-bit limit; JSON spec is wider |
| String value equal to the configured null placeholder | **Error** вҖ” collision; user must reconfigure via `$tomljson-null` directive or `TomlJsonOptions` |
| Strings that look like TOML literals (`"42"`, `"2026-05-04"`, `"true"`, `"inf"`) | Always quoted on output (`toml_writer` handles escaping) |
| Other JSON values (string, finite number, bool, array, object) | Direct emission via `toml_writer` primitives |

#### TOML вҶ’ JSON (decode)

| Concern | Handling |
|---|---|
| TOML string equal to the configured placeholder | Decode as `Json::Null` |
| Root-level table containing exactly one key `__root__` | Unwrap and return the inner value |
| Root-level `$tomljson-null` directive (standalone API only) | Capture the placeholder string from it; remove from output |
| TOML datetime (any of: `Date`, `Time`, `LocalDateTime`, `OffsetDateTime`) | Decode as `Json::String` using the canonical RFC 3339 form (`"2026-05-04"`, `"09:30:00"`, `"2026-05-04T09:30:00Z"`, etc.) |
| TOML float `+inf` / `-inf` | **Error** вҖ” invalid JSON; users wanting "no upper/lower bound" should omit the keyword instead |
| TOML float `NaN` | **Error** вҖ” invalid JSON; in JSON Schema, "no constraint" is expressed by absence, not NaN |
| TOML integer (i64) | Decode as `Json::Number` (i64-backed) |
| TOML finite float (f64) | Decode as `Json::Number` (f64-backed) |
| TOML bool, array, table | Recursively decode |
| Absent TOML key | Absent in JSON object вҖ” no special handling. **The TOML grammar has no syntax for "key with no value"; that's a parse error, not a null.** Distinct from JSON's `{ "key": null }` (key present, value null), which round-trips through the placeholder. |

The TOMLвҶ’JSON direction is the **dominant one** in mdvs: users author JSON Schema in TOML form, mdvs decodes it for internal validation. Both spike scripts validate this direction; `scripts/test_tomljson_decode.rs` covers it explicitly with 31 cases.

### Why we error on `+inf` / `-inf` / `NaN`

These values **could** survive in storage layers we use:
- Apache Parquet stores Float64 as IEEE 754, including non-finite values.
- Apache Arrow Float64 supports them in memory.
- LanceDB Float64 supports them (recent fix per [lancedb#3153](https://github.com/lancedb/lancedb/issues/3153)).

But they **cannot** survive validation:
- `serde_json::Number::from_f64(f64::NAN)` returns `None`. Documented behavior: *"Infinite or NaN values are not JSON numbers."*
- The `jsonschema` crate operates on `serde_json::Value`, so it can never see them.
- JSON Schema itself has no syntax for "no upper bound" except omitting `maximum`.

So even if `tomljson` silently allowed `+inf`, the validation pipeline would refuse it later with a worse error. Erroring at the TOMLвҶ’JSON boundary surfaces the problem immediately and points users at the idiomatic JSON Schema solution (omit the keyword).

### Other JSONвҶ”TOML edge cases (handled)

| Case | Behavior |
|---|---|
| Strings shaped like TOML literals (`"true"`, `"42"`, `"2026-05-04"`, `"inf"`) | Encoder always quotes; decoder parses as strings. No collision. |
| Hand-written unquoted TOML datetime (`default = 2026-05-04`) | Decoder converts `Toml::Datetime` to a JSON string. TOML form is normalized to quoted on re-encode. |
| Hand-written unquoted TOML local time (`default = 09:30:00`) | Same as datetime вҖ” converted to JSON string `"09:30:00"`. |
| Empty root schema `{}` | Encodes to empty TOML, decodes back to `{}`. |
| Heterogeneous arrays | TOML 1.1 spec explicitly allows `numbers = [0.1, 0.2, 1, 2]`. The `toml` crate honors this. |
| `serde_json` u64 within i64 range | Roundtrips losslessly via `Toml::Integer`. |
| f64 precision (subnormals, `f64::MIN_POSITIVE`, `f64::MAX`, `0.1 + 0.2`) | Rust `toml` crate uses RyЕ«-style shortest-roundtrip serialization вҖ” values roundtrip exactly. Concern raised in [toml-lang/toml#44](https://github.com/toml-lang/toml/issues/44) does not apply in practice. |
| Unicode in strings (`"cafГ© вҳ• вҖ” ж—Ҙжң¬иӘһ вҖ” рҹҡҖ"`) | Roundtrips losslessly. |
| Empty string values (`""`) | Roundtrip. |
| Strings with embedded newlines | Encoded as TOML multi-line basic strings (`"""..."""`); decoded back to JSON strings with `\n`. |
| TOML `+inf`, `-inf`, `nan` floats on decode | **Error.** See "Per-direction policy table" above вҖ” `serde_json::Number::from_f64` rejects them, and `jsonschema` never sees them, so erroring early is the only honest behavior. |
| TOML datetime on decode (any of 4 variants) | **JSON string** in canonical RFC 3339 form. JSON Schema uses strings + `format` for dates. |
| Absent TOML key vs. JSON `null` | TOML grammar has no syntax for "key with no value" (parse error). Absent TOML key вҶ’ absent JSON object key. JSON `key: null` round-trips through the placeholder string. |
| TOML comments | Lost on roundtrip. JSON has no comments. Acceptable вҖ” comments are author intent, not schema data. |

**Out of scope**: validation (delegated to `jsonschema`), schema authoring helpers, type generation.

### Prototype findings

Three `rust-script` prototypes validate the design end-to-end:

- `scripts/test_tomljson.rs` вҖ” encoding-rules baseline, 37 cases via a generic JSONвҶ”TOML Value walker. Establishes the canonical TOML form for every JSON Schema construct.
- `scripts/test_tomljson_writer.rs` вҖ” JSONвҶ’TOML (encode) spike, 18 cases via `toml_writer` dispatcher. Validates the chosen production approach for the encode direction.
- `scripts/test_tomljson_decode.rs` вҖ” TOMLвҶ’JSON (decode) spike, 31 cases. Explicitly exercises the per-direction policy table: all four datetime variants, +inf/-inf/NaN errors, placeholder substitution at scalar / array / nested positions, `__root__` unwrap, custom placeholder via `$tomljson-null`, inline tables, arrays-of-tables.

The encoding rules below are exercised across: basic types, constraints, `enum` heterogeneity (including `null`), `oneOf`/`anyOf`/`allOf`/`not`, `$defs`/`$ref`, `if`/`then`/`else`, deep nesting, `x-mdvs-*` extensions, boolean schemas, number fidelity, custom null placeholders via the `$tomljson-null` directive, placeholder-collision detection, strings shaped like TOML literals, unquoted TOML datetime input, empty root schema, integer-range boundaries (i64::MAX/MIN roundtrip; u64 > i64::MAX errors), and a realistic composite. All cases pass; canonical hand-written TOML for the composite decodes to the same JSON.

Decisions confirmed:

- **Null encoding** вҖ” JSON `null` is represented by a string placeholder (default `"__null__"`, configurable). The placeholder occupies the slot uniformly: `default = "__null__"`, `enum = [1, 2, "__null__"]`, `const = "__null__"`. Arrays stay homogeneous-looking instead of mixing in inline tables. See "Null encoding" section above for the directive form and embedded API.
- **Boolean-schema root** вҖ” top-level boolean schemas wrap under `__root__ = true|false`. Dunder convention matches the null sentinel.
- **Heterogeneous arrays** вҖ” `enum = [1, "two", true]` works directly; the [TOML 1.1 spec](https://toml.io/en/v1.1.0) explicitly permits mixed-type arrays and the `toml` crate honors this.
- **Number fidelity** вҖ” integer vs float distinction preserved through `serde_json::Number` (i64 вҶ” `Toml::Integer`, f64 вҶ” `Toml::Float`).
- **Integer range** вҖ” i64::MAX/MIN roundtrip exactly; values in `(i64::MAX, u64::MAX]` error on encode (TOML spec compliance).
- **String/literal disambiguation** вҖ” strings shaped like TOML literals (`"42"`, `"true"`, `"2026-05-04"`) always emitted as quoted strings. Hand-written unquoted TOML datetimes accepted on input (decoded to JSON strings).
- **TOMLвҶ’JSON datetime policy** вҖ” all four TOML datetime variants (`Date`, `Time`, `LocalDateTime`, `OffsetDateTime`) decode to JSON strings using their canonical RFC 3339 representation. JSON Schema represents dates as strings + `format` keyword anyway.
- **TOMLвҶ’JSON `+inf` / `-inf` / `NaN` policy** вҖ” error on decode. `serde_json::Number::from_f64` rejects them; the validation layer can never see them. Erroring early surfaces the limitation at the right place. Storage layers (Parquet, Arrow, LanceDB) support these values; the validation pipeline (jsonschema crate) does not. Users wanting "no upper/lower bound" should omit `maximum`/`minimum` per JSON Schema convention.
- **`nullable: bool` in mdvs is unrelated to TOML's data model** вҖ” it controls whether *YAML frontmatter values* may be null, not TOML config values. mdvs does not extend or reinterpret TOML grammar anywhere.
- **Reserved-looking keys** вҖ” `$schema`, `$id`, `$ref`, `$defs` auto-quote; `x-mdvs-*` (hyphens allowed in bare keys) does not need quoting.
- **`$defs` nesting** вҖ” `["$defs".address]` and deeper paths work as expected.

Open issues for the crate to handle (not blockers):

- **Array-of-tables sub-key emission** вҖ” `toml::to_string` emits `[anyOf.properties.kind]` (singular path under `[[anyOf]]`) which TOML semantics correctly bind to the most recent array element, but is visually confusing in hand-written form. The crate should prefer explicit `[[anyOf]]` re-headers and document the binding rule.
- **Key ordering** вҖ” `toml::to_string` sorts alphabetically. Acceptable for a canonical form (deterministic output) but loses input ordering. If we want to preserve author intent on roundtrip, switch to `toml_edit`.
- **Placeholder collision** вҖ” when the schema legitimately contains a string equal to the configured null placeholder, or a top-level key named `__root__`, the encoder errors and the user picks a different placeholder via `$tomljson-null` (standalone) or `TomlJsonOptions::null_placeholder` (embedded).
- **Verbose float emission** вҖ” the `toml` crate prints small/large floats in long decimal form rather than scientific notation (e.g., `f64::MIN_POSITIVE` renders as a 300+ digit decimal). Roundtrip-correct but unreadable. Mitigation: switch to `toml_edit` or post-process float emission for canonical output.

## Future crate split (post-TODO-0149, design context only)

The Wave A workspace introduces `tomljson` alongside `mdvs`. Once the three waves of TODO-0149 are stable, a further split is anticipated but **not in scope here** вҖ” captured as design context so subsequent work has a north star.

### Target end state (post-1.0)

```
crates/
в”ңв”Җв”Җ tomljson/           вҶҗ already in scope (Wave A)
в”ңв”Җв”Җ mdvs-validate/      вҶҗ future: scan, infer, schema, validate, preprocess
в”ңв”Җв”Җ mdvs-search/        вҶҗ future: chunk, embed, storage, backend, search
в””в”Җв”Җ mdvs/               вҶҗ future: CLI binary, depends on both
```

Mapping from the Wave-A-finished layout:

| Module today (`crates/mdvs/src/`) | Future home |
|---|---|
| `discover/`, `schema/`, `preprocess/` | `mdvs-validate` |
| `index/`, `search.rs` | `mdvs-search` |
| `cmd/`, `main.rs` | `mdvs` (CLI binary) |
| `lib.rs` | reexport facade in `mdvs` if needed |

### Why this split (over the alternatives)

Three split axes were considered:

1. **Engine + CLI** (`mdvs-core` + `mdvs`): standard Rust idiom (cargo, ripgrep, fd). Right answer eventually but creates public API surface for which there is no second consumer today.
2. **Feature split** (`mdvs-validate` + `mdvs-search` + `mdvs` CLI): the chosen target. Validation and search have distinct user audiences; validation has zero ML/embedding deps and stands alone as "JSON Schema validation for markdown directories." Both libraries get a real consumer (the CLI binary), validating their APIs.
3. **Provider extraction** (separate crates per backend / per embedder): too granular for the codebase size; the existing enum-based dispatch already gives this at the type level.

The feature split is the most defensible because:

- `mdvs-validate` is a coherent product on its own вҖ” likely audience overlap with `tomljson` users.
- ML/embedding dependency cost only flows to consumers that actually use search.
- LanceDB backend (TODO-0016) and additional embedding providers (TODO-0017) live naturally in `mdvs-search` without bloating validation.

### Triggers for the split

Any of:
- An external user / issue asks "can I use mdvs's validation as a library?"
- We decide to publish `mdvs-validate` standalone for discoverability.
- A second consumer of the engine emerges (server, language extension, IDE plugin).

Until at least one trigger fires, keeping everything in one `mdvs` crate avoids paying boundary-design cost for benefit we don't yet have.

### Out of scope for TODO-0149

This split is a **separate future TODO**, not part of TODO-0149. The Wave A/B/C work in this TODO assumes the current single-`mdvs` layout (with `tomljson` extracted alongside). The split decision should be made when triggered, not pre-committed.
