# TODO-0149: JSON Schema export and per-value validation delegation (two tracks)

## Summary

Two independent features, previously bundled as one, now split:

- **Track A — Export:** produce a JSON Schema document from mdvs.toml so external tools can consume the schema. Internal validator unchanged.
- **Track B — Delegate:** replace our `ConstraintKind::validate_value` with calls into the `jsonschema` crate. Nothing exported.

Each track has different costs and buys different value. They can be pursued independently, together, or not at all. This TODO holds the design for both and the evaluation criteria; follow-up TODOs implement whichever track(s) are accepted.

## Why unbundled

The original framing made JSON Schema "the canonical representation" of `[fields]`, which forced both tracks at once. There is no structural reason to couple them — the internal validator representation can be anything (keep today's `Constraints` struct) while the external artifact is JSON Schema.

|                                          | Track A only | Track B only | Both     |
| ---------------------------------------- | ------------ | ------------ | -------- |
| Dep added                                | `schemars` (small) | `jsonschema` (big) | `jsonschema` + translator |
| Internal validator                       | unchanged    | rewritten    | rewritten |
| External tools can read the schema       | yes          | no           | yes      |
| Lines of mdvs code removed               | ~0           | ~200         | ~200     |
| "Standards-adherent" positioning         | yes          | no           | yes      |

Track A buys interchange / positioning cheaply, no validator churn. Track B buys code reduction and free per-value features (`pattern`, `min_length`), but only pays off if the library's error messages can match today's quality and the dep weight is acceptable.

## Track A — Export JSON Schema

### Goal

`mdvs export-schema [path] [--output-file schema.json]` produces a JSON Schema document representing the `[fields]` section. External tools consume it; mdvs's own validator is untouched.

### Design

- Use `schemars` (derive-based) or a hand-written translator module. Prefer hand-written if `schemars` forces awkward type shapes.
- Standard keywords only for per-value constraints: `type`, `enum`, `minimum`/`maximum`, `pattern`, `minLength`/`maxLength`.
- Path-aware rules ride as extensions: `x-mdvs-allowed`, `x-mdvs-required`. Tools that don't understand them ignore them silently.
- The export represents the **strict** contract. Users running mdvs in lenient mode (current default) get per-value validation that's a superset of what the exported schema enforces — this is documented on the export command.

### What's in scope, what isn't

In: `[fields].field[*]` entries — name, type, constraints, nullable, allowed, required.
Out: `[scan]`, `[check]`, `[build]`, `[search]`, `[update]`, `[embedding_model]`, `[chunking]`, `[fields].ignore`, `[fields].max_categories`, `[fields].min_category_repetition`. These are mdvs configuration, not schema.

### Open questions

1. **Draft version** — 2020-12 (modern, `$vocabulary`) vs draft-07 (widest tool compat)?
2. **Extension naming** — `x-mdvs-*` (OpenAPI convention, simple) vs formal `$vocabulary` URI (more correct, more ceremony)?
3. **`nullable` encoding** — `"type": ["string", "null"]` (canonical) vs `"nullable": true` (OpenAPI)?
4. **Import direction (optional stretch):** can mdvs read a JSON Schema and produce a `mdvs.toml`? The import path is where we discover what mdvs genuinely can't represent (`oneOf`, `$ref`, `additionalProperties: false`). Worth a failure-mode probe even if we don't ship import.

### Evaluation criteria

- A sample export from `example_kb/mdvs.toml` reads cleanly and round-trips through a third-party JSON Schema validator (e.g. `ajv`) against an example frontmatter document.
- `schemars` dep weight is acceptable (or hand-written translator avoids the dep entirely).

## Track B — Delegate per-value validation

### Goal

Replace `ConstraintKind::validate_value()` with calls into the `jsonschema` crate. Today's ~200 lines of per-constraint validation become a thin adapter that (1) translates our `Constraints` to a `jsonschema::Validator` at load time, (2) coerces values per mdvs's leniency rules (String-is-top, Int→Float), (3) calls the validator, (4) maps library errors back to `FieldViolation`.

### Design

```
┌───────────────────────────────────────────────────┐
│ 1. [mdvs] Type check with leniencies              │
│ 2. [mdvs] Coerce value to canonical form          │
│    (integer 1 → "1" for String field, etc.)       │
│ 3. [jsonschema] Strict validation                 │
└───────────────────────────────────────────────────┘
```

- Leniency stays in the pre-coercion layer; the library only sees post-coercion values.
- `Constraints::conflicts_with()` and `validate_for_type()` stay (JSON Schema has no opinion on these).
- Error mapping: preserve `OutOfRange`, `InvalidCategory`, etc. as distinct variants by inspecting `jsonschema` error output. If mapping is lossy, collapse to a single `ConstraintFailed` and accept the regression.
- The internal representation of `Constraints` stays as-is in mdvs.toml — translation happens at validator-construction time, not at parse time.

### Open questions

1. **Error message quality** — is the library's error structure rich enough to reconstruct messages like `"rating: got 11, expected min=1, max=5"`? This is the make-or-break question.
2. **Violation kinds** — preserve distinct kinds or collapse to one?
3. **Binary size / dep impact** — `jsonschema` pulls `url`, `regex`, `fancy-regex`. Check if the regex engine is configurable.

### Evaluation criteria

- Binary size delta acceptable (target: measure, then decide).
- Error messages for 10 representative cases match or exceed current quality (side-by-side diff attached to the prototype).
- Test suite passes unchanged.

## Constraint mapping: mdvs ↔ JSON Schema

This map drives both tracks. For export (A), it shows what JSON Schema keyword each mdvs constraint emits. For delegation (B), it shows what the adapter has to build at validator-construction time.

### Per-value constraints

| mdvs constraint | mdvs applies to | JSON Schema keyword(s) | On scalar | On array | Notes |
| --- | --- | --- | --- | --- | --- |
| `categories` | String, Integer, Array(String), Array(Integer) | `enum` | `{"type": "string", "enum": [...]}` | `{"type": "array", "items": {"enum": [...]}}` (per-item) | JSON Schema `enum` has no type restriction; we restrict on the mdvs side via `validate_for_type`. |
| `min` / `max` | Integer, Float, Array(Integer), Array(Float) | `minimum`, `maximum` | `{"type": "integer", "minimum": X, "maximum": Y}` | `{"type": "array", "items": {"minimum": X, "maximum": Y}}` (per-item) | Inclusive in both systems. Exclusive bounds would use `exclusiveMinimum`/`exclusiveMaximum` if we ever add them. |
| `min_length` / `max_length` (TODO-0010) | String, Array | **different keywords per type** | `minLength`/`maxLength` (string) | `minItems`/`maxItems` (array) | Same mdvs constraint → different JSON Schema keywords depending on FieldType. Export translator must dispatch on type. |
| `pattern` (TODO-0145) | String | `pattern` | `{"type": "string", "pattern": "..."}` | n/a | Regex flavor mismatch: mdvs would use `regex` crate (RE2-like), JSON Schema spec mandates ECMA-262, `jsonschema` crate uses `fancy-regex`. Pick one flavor and document it. |
| `nullable: bool` | any type | `"type": ["T", "null"]` or `"nullable": true` | `{"type": ["string", "null"]}` | `{"type": ["array", "null"]}` | Open question #3 — pick encoding. `["T","null"]` is standard 2020-12; `"nullable": true` is OpenAPI shorthand and shorter. |

### Type-dispatch contract

- **JSON Schema model:** keywords that don't apply to the instance type are silently ignored. `{"type": "string", "minimum": 5}` is legal and does nothing.
- **mdvs model:** `validate_for_type` rejects misapplied constraints at config-load time.
- **Implication for Track B:** `validate_for_type` stays — it's an mdvs-level UX check that runs before the library is ever invoked. The library never sees a misapplied constraint.
- **Implication for Track A:** because mdvs pre-rejects misapplied combinations, exported schemas are correct by construction — no dangling `minimum` on strings will appear.

### Pairwise conflict contract

- **JSON Schema model:** mutually contradictory constraints are legal but unsatisfiable. `{"minimum": 10, "maximum": 5}` validates nothing.
- **mdvs model:** `conflicts_with` rejects such combinations at config-load time (e.g. duplicate `categories`, `categories` + `range` on the same field).
- **Implication for both tracks:** `conflicts_with` stays on the mdvs side. Not expressible in JSON Schema, not something the library enforces.

### Path-aware rules (not per-value)

| mdvs construct | JSON Schema representation |
| --- | --- |
| `allowed: [glob, ...]` | `x-mdvs-allowed: [glob, ...]` (extension) |
| `required: [glob, ...]` | `x-mdvs-required: [glob, ...]` (extension) |

Third-party validators silently ignore extensions. Exported schemas intended for external consumption document that path-aware rules are not enforced outside mdvs.

### Things mdvs cannot (yet) represent on import

If Track A also supports import of third-party JSON Schemas, these constructs have no mdvs equivalent today and would need to be rejected or ignored with a warning:

- `oneOf`, `anyOf`, `allOf` (discriminated unions / composition)
- `$ref` (schema references)
- `additionalProperties: false` (closed-object semantics — mdvs uses `allowed` globs for a similar effect but scoped to paths, not object keys)
- `format` other than what we map to our Date type (once TODO-0007 lands)
- `dependencies`, `if`/`then`/`else`

## Deliverable: a half-day spike on Track B

Start here because Track B has the harder unknowns (error quality, dep weight) and Track A depends on nothing in Track B. If Track B fails its evaluation criteria, Track A can still proceed.

1. Add `jsonschema` as a dep, `cargo build --release`, record binary size delta.
2. Pick **one** constraint (`range`, since it has two fields so the reduction isn't trivial).
3. Delegate that one constraint through the adapter pattern above.
4. Generate 10 error cases — mix of valid and invalid values, edge cases (null, type mismatch, at-boundary). Print current-mdvs output and `jsonschema` output side-by-side. Attach to this TODO.
5. Decide: commit to Track B (spin off follow-up TODO), defer, or reject.

Track A can be prototyped in parallel if there's appetite — it's independent.
