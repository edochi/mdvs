# TODO-0149: Bidirectional mdvs.toml ↔ JSON Schema and per-value validation delegation

## Summary

Establish JSON Schema (with a mdvs-specific vocabulary extension for path-aware rules) as the canonical representation of the `[fields]` section, with a **bidirectional, lossless** translation to and from mdvs.toml. Delegate per-value validation (type, enum, range, length, pattern) to the `jsonschema` library. This gives mdvs an industry-standard interchange format, eliminates reimplementation of JSON Schema semantics, and lays the groundwork for a future engine extraction.

## Motivation

Our per-value validation reimplements what mature JSON Schema libraries already provide — ~200 lines of Rust that will grow as we add `pattern`, `min_length`/`max_length`, etc. We're reinventing a well-solved problem.

But the real reframing is bigger than "use a library." Three observations:

1. **TOML is good for authoring** (humans and agents edit it directly), but it's not an interchange format.
2. **JSON Schema is the industry-standard interchange format** for value validation, and we already express every per-value concept it has.
3. **Path-aware rules** (`allowed`/`required` globs) are genuinely novel — no JSON Schema library addresses them — but they can ride as annotations without invalidating the standard.

If we treat mdvs.toml and JSON Schema as **two views of the same schema**, with a bidirectional lossless translation, we get:

- **Interchange** via JSON Schema for any tool that consumes it
- **Authoring UX** via TOML for humans
- **Engine extraction path** — the engine operates on the canonical representation, the CLI is a product layer on top
- **Free per-value validation** — delegate to `jsonschema`, no reimplementation

## Core constraint: bidirectional losslessness

Every mdvs.toml `[[fields.field]]` entry has a faithful JSON Schema representation, and every JSON Schema property (with our vocab) has a faithful mdvs.toml representation. A round-trip preserves all information.

**Example:**

```toml
[[fields.field]]
name = "rating"
type = "Integer"
allowed = ["projects/**"]
required = ["projects/alpha/**"]
nullable = false

[fields.field.constraints]
min = 1
max = 5
```

```json
{
  "rating": {
    "type": "integer",
    "minimum": 1,
    "maximum": 5,
    "x-mdvs-allowed": ["projects/**"],
    "x-mdvs-required": ["projects/alpha/**"]
  }
}
```

- Standard JSON Schema keywords (`type`, `minimum`, `maximum`, `enum`, `pattern`, ...) for per-value constraints
- `x-mdvs-*` extension keywords for path-aware rules
- A third-party JSON Schema tool reads the schema and can perform per-value validation, silently ignoring the extensions
- mdvs reads the same schema and performs both per-value and path-aware validation

## Leniencies stay — they're pre-coercion, not schema

Current mdvs validation has two "leniency" rules that are **inference-correctness requirements, not smells**:

- **"String is the top type"** — a String-typed field accepts any value. Inference widens mixed types to String; validation must then accept the values that caused widening.
- **"Float accepts Integer"** — YAML doesn't distinguish `5` from `5.0`.

These operate at the **type-check / pre-coercion** layer, not at the constraint layer. The JSON Schema we generate is strict (`"type": "string"`), but mdvs's validator coerces values into canonical form before delegating:

```
┌───────────────────────────────────────────────────┐
│ 1. [mdvs] Type check with leniencies              │
│ 2. [mdvs] Coerce value to canonical form          │
│    (integer 1 → "1" for String field, etc.)       │
│ 3. [jsonschema] Strict validation against schema  │
└───────────────────────────────────────────────────┘
```

The library never sees a leniency case — it receives post-coercion values. The exported JSON Schema represents the declared contract; the leniencies are how mdvs bridges raw YAML to that contract. Third-party tools get strict validation, which is what they expect.

(A future "strict mode" — TODO-0150 — will skip step 2 and delegate raw values to the library.)

## Architecture

### Layers

```
┌──────────────────────────────────────────────────┐
│ CLI / product: mdvs.toml authoring, --with, etc. │
├──────────────────────────────────────────────────┤
│ TOML ↔ JSON Schema translation (bidirectional)   │
├──────────────────────────────────────────────────┤
│ Canonical representation: JSON Schema + vocab    │
├──────────────────────────────────────────────────┤
│ Engine: inference + path-aware + per-value check │
│   - inference: produces canonical rep            │
│   - path-aware: reads x-mdvs-* extensions        │
│   - per-value: delegates to jsonschema crate     │
└──────────────────────────────────────────────────┘
```

### mdvs vocabulary (extension keywords)

Defined on per-field schemas:

- `x-mdvs-allowed: [glob, ...]` — globs where this field is allowed to appear
- `x-mdvs-required: [glob, ...]` — globs where this field MUST appear

(Future: `x-mdvs-*` additions as more path-aware behavior is needed.)

### What doesn't translate

Some mdvs.toml content is tool-configuration, not schema — it stays TOML-only:

- `[scan]` — glob, include_bare_files, skip_gitignore (scan configuration)
- `[check]`, `[build]`, `[search]`, `[update]` — auto_update, auto_build, etc. (pipeline config)
- `[embedding_model]`, `[chunking]` — build-time configuration
- `[fields].ignore` — this is a validator-behavior list, not a schema element. Could go as a top-level `x-mdvs-ignore` in the JSON Schema, or stay TOML-only.
- `[fields].max_categories`, `[fields].min_category_repetition` — inference heuristic tuning, not schema.

These are mdvs configuration, not mdvs *schema*. The JSON Schema export represents the schema only.

## Proposed file changes

- `src/schema/constraints/json_schema.rs` — new module:
  - `to_json_schema(&Constraints, field_type: &FieldType) -> serde_json::Value` — single-field translation
  - `from_json_schema(&serde_json::Value) -> (Constraints, FieldType)` — reverse translation
  - Roundtrip tests
- `src/schema/config.rs` — add methods:
  - `MdvsToml::to_json_schema(&self) -> serde_json::Value` — full `[fields]` to a JSON Schema document
  - `MdvsToml::fields_from_json_schema(&serde_json::Value) -> FieldsConfig` — reverse
  - Roundtrip tests
- `src/schema/constraints/mod.rs`:
  - Rework `ConstraintKind::validate_value()` to delegate to `jsonschema` for the standard parts
  - Keep `conflicts_with()` (pairwise constraint conflict rules that JSON Schema has no opinion on)
  - Keep `validate_for_type()` (applicability checks)
- `src/cmd/` — new command: `mdvs export-schema [path] [--output-file schema.json]`
- `Cargo.toml` — add `jsonschema` dependency (measure binary size impact before/after)

## Open questions to resolve in the prototype

1. **JSON Schema draft version** — 2020-12 (latest, formal `$vocabulary` support) vs draft-07 (widest tool compatibility)?
2. **Extension naming** — `x-mdvs-*` (OpenAPI convention) vs formal `$vocabulary` URI?
3. **`nullable` encoding** — `"type": ["string", "null"]` (JSON Schema canonical) vs `"nullable": true` (OpenAPI / shorter)?
4. **Error message quality** — is `jsonschema`'s error structure good enough to build our friendly messages (`"rating: got 11, expected min=1, max=5"`), or do we wrap the library's output with our own context?
5. **Violation kinds** — do we preserve `OutOfRange`, `InvalidCategory` as distinct kinds after delegation (by inspecting library output), or collapse to a single `ConstraintFailed`?
6. **Binary size / dep impact** — `jsonschema` pulls `url`, `regex`, `fancy-regex`. Measure before/after.
7. **Import scenario** — can mdvs read a JSON Schema with our vocab and produce a mdvs.toml? Useful for users migrating from JSON Schema sources.

## Deliverable

Not one big refactor — a **prototype + decision**:

1. Implement the bidirectional translation for current constraints (`categories`, `min`/`max`) + path-aware extensions.
2. Prototype validation delegation for one constraint (start with `categories`).
3. Measure: dep weight, error message quality, test coverage.
4. Compare: side-by-side mdvs.toml + generated JSON Schema + round-trip.
5. Decide: commit to full migration (spin off follow-up TODO), defer, or reject.
