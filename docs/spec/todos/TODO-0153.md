# TODO-0153: General-purpose context-aware validation engine (potential extraction)

## Summary

Captures the design context for a hypothetical future Rust crate that generalizes mdvs's validation pipeline: JSON Schema 2020-12 validation + a three-stage preprocessor pipeline (with built-in variants and a Lua escape hatch) + **context-aware custom keywords** that can express conditional rules based on out-of-band context. mdvs would consume this engine and register a single `path_glob` keyword to recover its current path-scoping semantics. **Deferred** — captured here as a north star, not a commitment.

## Why this exists as a TODO

During TODO-0149's design discussion, we asked whether the validation pipeline mdvs is building (jsonschema + path-scoping + preprocessors + Lua) is novel enough to extract as a standalone crate. Initial answer: no, it's mdvs-specific. Refined answer: **the engine itself could be genuinely general, IF the path-scoping is reframed as a special case of "context-aware keywords"**.

The realization came from auditing what mdvs's path-scoping actually does at runtime: a glob match against a string (the filepath). Internally, `globset` compiles globs to regex; the runtime check is `regex.is_match(&filepath)`. The novelty in mdvs about paths is **the inference algorithm** (`DirectoryTree`, `GlobMap`, glob-collapse), not the runtime validation.

Once you accept that path-scoping is just "match a string in the validation context against a pattern declared in the schema", the same mechanism cleanly subsumes:

- **Multi-tenant validation** — `context = {tenant_id: "acme"}`, schema declares `x-required-when: {tenant_id: "acme"}`.
- **Environment-conditional validation** — `context = {env: "prod"}`, fields required only in production.
- **API versioning** — `context = {api_version: "v2"}`, deprecation/migration rules.
- **Feature-flag-conditional schemas** — `context = {flag: "beta"}`, conditional fields.
- **mdvs's path-scoping** — `context = {path: "docs/published/launch.md"}`, fields scoped via glob keyword.

## The Rust ecosystem gap

The Rust validation ecosystem splits into two camps that don't cover this shape:

| Approach | Crates | What it covers |
|---|---|---|
| **Schema-as-Rust-type** | `validator`, `garde`, `nutype`, `serde` + custom impls | known-at-compile-time schemas, derived from struct definitions |
| **Schema-as-data, validation-only** | `jsonschema` | JSON Schema 2020-12 validation, no preprocessing, no custom-keyword runtime extension hooks |

There is no good Rust answer for **"schema-as-data + preprocessing pipeline + context-aware custom keywords"**. That's the Pydantic v2 shape — Pydantic explicitly supports a `context={...}` parameter passed alongside the data, accessible from custom validators.

The Rust audience for this is genuinely narrower than the Python audience for Pydantic (Rust has compile-time types; the Pydantic motivation of "runtime type safety in a dynamic language" doesn't apply). But the niche is real: **anyone validating data whose schema is itself data**, like config validators, document validators, multi-tenant systems, plugin-loaded schemas, and tools (like mdvs) where users author schemas at runtime.

## Engine sketch

```rust
pub struct Engine {
    global: jsonschema::Validator,           // standard JSON Schema validation
    custom_keywords: KeywordRegistry,        // user-registered context-aware keywords
    preprocess: Pipeline,                    // 3-stage: key-rename, value-transform, document-transform
}

pub struct Context(HashMap<String, String>); // opaque key-value bag

pub trait ContextKeyword {
    /// The keyword name as it appears in the schema (e.g., "x-path-glob").
    fn name(&self) -> &str;

    /// Decide whether the field this keyword decorates is "active" given the context.
    /// Used for required/forbidden-field semantics.
    fn applies(&self, args: &serde_json::Value, ctx: &Context) -> bool;

    /// Optional: validate the value itself given context.
    /// Most keywords only need `applies()`; this is for richer cases.
    fn validate(&self, _value: &serde_json::Value, _args: &serde_json::Value, _ctx: &Context)
        -> Result<(), KeywordError> { Ok(()) }
}

impl Engine {
    pub fn builder() -> EngineBuilder;
    pub fn validate(&self, instance: &serde_json::Value, ctx: &Context) -> Vec<Violation>;
}

// mdvs registers a single keyword:
let engine = Engine::builder()
    .with_schema(canonical_schema)?
    .with_keyword(PathGlobKeyword::new())   // 30-line struct
    .with_preprocessors(builtins())
    .build()?;
```

The runtime pattern (already designed in TODO-0149) is the partition + per-context overlay approach: strip context-aware keywords at compile time, hold them aside, synthesize a per-context overlay schema each call, validate against (global ∧ overlay). That pattern is the engine's implementation; it isn't mdvs-specific.

## What mdvs would look like as a consumer

After the engine exists and mdvs migrates to it:

```
crates/
├── tomljson/                 (Wave A)
├── <engine>/                 (this TODO, deferred)
│   ├── jsonschema 2020-12 validation
│   ├── preprocessor pipeline + Lua hooks
│   ├── context-aware keyword registry
│   └── error mapping with structured locations
└── mdvs/
    ├── inference algorithm  ← the real mdvs novelty
    ├── path_glob keyword    ← ~30 LOC, consumed by engine
    ├── frontmatter-specific preprocessors (if any)
    ├── ingest layer (filesystem + YAML→JSON)
    ├── search layer (chunk + embed + storage + query)
    └── cli layer
```

mdvs's special parts shrink dramatically. The inference algorithm becomes the centerpiece. Validation becomes a library call.

## Why this is C, not B

Three paths were considered:

| Path | Description | Trade-off |
|---|---|---|
| A | TODO-0152 only — internal layering, no extraction | small commitment, clean internals, no general-purpose contribution |
| B | Build the engine as a sibling crate during Wave B; mdvs consumes it from day one | most ambitious; doubles surface area before mdvs has audience |
| **C (chosen)** | **Internal layering now (TODO-0152); extract the engine post-mdvs-1.0 if/when triggered** | safest; validates mdvs's audience first; extraction becomes obvious if/when concrete demand exists |

We chose C: TODO-0152 lands first, the engine concept lives in this TODO as a captured option, no commitment.

## Triggers for promoting this from "deferred" to "active"

Any one of:

1. **External demand**: an issue, blog comment, or upstream PR asking "can I use mdvs's validation as a library?" — concrete evidence that someone wants the engine separately from mdvs.
2. **Second consumer in our own work**: we end up building a second tool (config validator, document linter, etc.) that wants the same engine. Two consumers force the API into honesty.
3. **mdvs hits a meaningful audience**: post-1.0, with stars/users/issues. At that point spinning out the engine is a discoverability move, not a speculative one.
4. **Pydantic-equivalent gap is filled by someone else**: if a "rust-validate-with-context" crate appears and is good enough, this TODO can be closed with `superseded`. mdvs would adopt the external crate instead.

## Out of scope

- The engine's actual implementation. This TODO documents the concept; implementation is its own multi-month project, gated on the triggers above.
- mdvs's adoption of the engine. Would be a separate migration TODO once the engine exists.
- Naming the engine. Candidates floated during discussion: `valid8`, `schemata`, `rusticate`, `pydantically`. Settle this only when the work begins.

## Acceptance criteria

This TODO has no acceptance criteria — it's a captured design idea, not a planned piece of work. It's "complete" when one of the triggers fires and a successor TODO is created (which would then have its own acceptance criteria).
