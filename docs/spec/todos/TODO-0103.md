# TODO-0103: Validate config invariants on mdvs.toml load

## Summary

Two documented invariants in `mdvs.toml` are assumed but not enforced. Hand-edited configs can silently violate them. Add validation that catches both on config load.

## Details

### Invariant 1: required ⊆ allowed

Every glob in a field's `required` list must also be covered by its `allowed` list. The inference algorithm guarantees this by construction, but manual edits can break it. Currently no check exists — a field with `required = ["blog/**"]` and `allowed = ["notes/**"]` is silently accepted.

### Invariant 2: ignore and [[fields.field]] are mutually exclusive

A field name cannot appear in both the `[fields].ignore` list and as a `[[fields.field]]` entry. Currently, if a field is in both, the ignore check silently wins during validation and the `[[fields.field]]` entry is dead config. Should be an error.

### Where to add

Add a `validate_config()` function (or extend config loading) that checks both invariants. Call it from every command that reads `mdvs.toml` — check, build, update, search, info. The existing `check_reserved_names()` in `src/pipeline/write_config.rs` is a pattern to follow.

### Expected behavior

- Hard error with a clear message on violation
- Catches problems early, before any scanning/validation/building
