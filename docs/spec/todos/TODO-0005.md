# TODO-0005: Differentiate null from absent in check

## Summary

Currently null frontmatter values (`field:` with no value) are treated identically to absent fields in check. In the future, null and absent should be distinguishable — e.g. a field that is present-but-null might have different semantics than a field that is entirely missing.

## Details

Current behavior (implemented as a temporary measure):
- Null values are skipped during type/allowed validation (no WrongType or Disallowed)
- Null values are treated as absent for required validation (trigger MissingRequired)

Future behavior should allow the schema to express whether null is acceptable for a field, separately from whether the field can be absent. This may involve a `nullable` flag on `[[fields.field]]` or similar.
