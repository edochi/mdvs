# TODO-0078: Structured error output for all commands

## Summary

Rework command output to follow a uniform pipeline model. Every command is a sequence of processing steps, each producing a typed result. The command output separates the processing record (what steps ran) from the final result. Errors are contextualized by the step where they occurred. Both text and JSON formats render consistently.

## Details

### Core abstractions

#### ProcessingStep

A record of a single pipeline step's execution. No context field — step-specific metadata lives inside `T`.

```rust
struct ProcessingStep<T: StepOutput> {
    elapsed_ms: u64,
    output: T,
}
```

#### StepOutput trait

Every step output type implements `StepOutput` for uniform text rendering:

```rust
trait StepOutput: Serialize {
    fn format_line(&self) -> String;  // e.g. "Scanned 5 files"
}
```

`ProcessingStepResult<T: StepOutput>` renders any step as a line without the command knowing the details:
- `Completed` → `step.output.format_line()` + elapsed_ms
- `Failed` → step name + error message
- `Skipped` → step name + "skipped"

This avoids repetitive per-command rendering code. Adding a new step just requires implementing `StepOutput`.

#### ProcessingStepResult

The three states a step can be in:

```rust
pub enum ProcessingStepResult<T: StepOutput> {
    Completed(ProcessingStep<T>),
    Failed(ProcessingStepError),
    Skipped,
}
```

- **Completed** — the step ran and produced output. Validation violations are *output*, not failure.
- **Failed** — the step ran but hit an actual error. No typed output `T`, just an error.
- **Skipped** — the step didn't run because a previous step's outcome made it unnecessary.

#### ProcessingStepError

```rust
struct ProcessingStepError {
    kind: ErrorKind,    // User | Application
    message: String,
}
```

Two error categories (HTTP analogy):
- **User (4xx)** — bad input: config not found, model mismatch, invalid `--where` syntax
- **Application (5xx)** — unexpected internal failure: I/O errors, parquet corruption

### Command output pattern

Every command has two layers:

```rust
struct BuildProcessOutput {
    scan: ProcessingStepResult<ScanOutput>,
    validate: ProcessingStepResult<ValidateOutput>,
    classify: ProcessingStepResult<ClassifyOutput>,
    embed: ProcessingStepResult<EmbedFilesOutput>,
    write_index: ProcessingStepResult<WriteIndexOutput>,
}

struct BuildCommandOutput {
    process: BuildProcessOutput,
    result: Option<BuildResult>,  // None if pipeline didn't complete
}
```

- **Compact success**: only `result` is rendered
- **Verbose success**: `process` steps + `result`
- **Failure (compact or verbose)**: `process` steps up to the failure point, `result` is `None`

### Pipeline control

The pipeline control is NOT in the step — it's in the command's sequencing logic. The step is just a record of "what happened." The command decides the pipeline semantics: which step outcome is terminal, which is pass-through.

The same `ValidateOutput` with violations means "here's your result" in check, but "stop the pipeline" in build. The step doesn't know this — the command does.

### Flat pipelines — no nesting

Commands are always flat sequences of steps. Init and update do NOT embed a `BuildCommandOutput` — they flatten all build steps into their own pipeline:

```
init: scan → infer → write_config → validate → classify → load_model → embed → write_index
build: read_config → scan → validate → classify → load_model → embed → write_index
```

This means init and build share step *functions* but compose them independently. Both call `run_scan()`, `run_validate()`, etc. — the command just wires them together.

### Code structure

Steps are first-class modules. Each step module owns its run function, output struct, and `StepOutput` impl. Commands become thin orchestration layers.

```
src/
├── pipeline/
│   ├── mod.rs          # ProcessingStep, ProcessingStepResult, StepOutput trait
│   ├── scan.rs         # run_scan(), ScanOutput
│   ├── validate.rs     # run_validate(), ValidateOutput
│   ├── infer.rs        # run_infer(), InferOutput
│   ├── classify.rs     # run_classify(), ClassifyOutput
│   ├── load_model.rs   # run_load_model(), LoadModelOutput
│   ├── embed.rs        # run_embed_files(), run_embed_query(), EmbedFilesOutput, EmbedQueryOutput
│   ├── write_index.rs  # run_write_index(), WriteIndexOutput
│   ├── read_index.rs   # run_read_index(), ReadIndexOutput
│   └── ...
├── cmd/
│   ├── build.rs        # pipeline composition + BuildCommandOutput
│   ├── check.rs        # pipeline composition + CheckCommandOutput
│   └── ...
```

The existing `src/discover/`, `src/index/`, `src/search.rs` modules still hold the core domain logic. Pipeline step modules call into them and wrap results in `ProcessingStepResult`.

### Error handling

All anticipated errors are caught explicitly inside step functions and returned as `ProcessingStepResult::Failed`. Commands return `anyhow::Result<*CommandOutput>` as a last safety net for truly unexpected failures (panics, OOM), but this should ideally never fire.

### Output examples

**Build — compact, success:**
```
Built index — 5 files, 12 chunks (full rebuild)

╭──────────────────────────────────────╮
│ embedded    5 files    12 chunks     │
╰──────────────────────────────────────╯
```

**Build — verbose, success:**
```
Scanned 5 files
Validated 5 files — no violations
Loaded model minishlab/potion-base-8M (1200ms)
Built index — 5 files, 12 chunks (full rebuild)

╭──────────────────────────────────────╮
│ embedded    5 files    12 chunks     │
╰──────────────────────────────────────╯
```

**Build — compact, validation violations:**
```
Scanned 5 files
Validated 5 files — 1 violation(s)

╭──────────────────────────────────────╮
│ "tags"   MissingRequired   1 file    │
╰──────────────────────────────────────╯
```

**Build — compact, infra error:**
```
Scanned 5 files
Validated 5 files — no violations
Loading model — failed: could not find "minishlab/potion-base-8M"
```

**Build — JSON, validation violations:**
```json
{
  "process": {
    "scan": { "status": "completed", "output": { "files": 5 } },
    "validate": {
      "status": "completed",
      "output": {
        "files_checked": 5,
        "violations": [{ "field": "tags", "kind": "MissingRequired", ... }]
      }
    },
    "classify": { "status": "skipped" },
    "embed": { "status": "skipped" },
    "write": { "status": "skipped" }
  },
  "result": null
}
```

**Build — JSON, infra error:**
```json
{
  "process": {
    "scan": { "status": "completed", "output": { "files": 5 } },
    "validate": { "status": "completed", "output": { "files_checked": 5, "violations": [] } },
    "classify": { "status": "completed", "output": { ... } },
    "embed": { "status": "failed", "error": { "kind": "user", "message": "could not find model" } },
    "write": { "status": "skipped" }
  },
  "result": null
}
```

### Exit codes

- 0: success (pipeline completed, result present)
- 1: normal result with violations (pipeline completed, check/build found violations)
- 2: user error (4xx)
- 3: application error (5xx)
