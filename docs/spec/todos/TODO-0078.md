# TODO-0078: Structured error output for all commands

## Summary

All commands should produce structured output on errors, both in text and JSON format. Currently, errors go through `anyhow::bail!()` as unstructured text to stderr — in JSON mode, no JSON is produced at all.

## Details

### Error taxonomy (HTTP analogy)

Three tiers of command outcomes:

1. **Normal results (2xx)** — the command did its job. Validation violations are normal output, not errors. Check finding 3 violations is a successful check. Build aborting due to violations is a successful validation that prevented a bad build. These use the existing `CommandOutput` structs.

2. **User errors (4xx)** — the user provided bad input. Examples: `mdvs.toml` not found, no markdown files found, model mismatch between config and parquet, invalid `--where` syntax, invalid flag combinations. These are currently `anyhow::bail!()` calls with descriptive messages.

3. **Application errors (5xx)** — unexpected internal failures. Examples: I/O errors reading files, parquet corruption, serialization failures. These are currently `?`-propagated errors.

### Step-based output model

Multi-step commands (build, init, update) should expose their pipeline steps in the output. This contextualizes errors — you see exactly which phase failed and what succeeded before it.

**Compact, success** — just the final result (steps are invisible):
```
Built index — 5 files, 12 chunks (full rebuild)

╭──────────────────────────────────────╮
│ embedded    5 files    12 chunks     │
╰──────────────────────────────────────╯
```

**Verbose, success** — every step visible:
```
Scanned 5 files
Validated 5 files — no violations
Loaded model minishlab/potion-base-8M (1200ms)
Built index — 5 files, 12 chunks (full rebuild)

╭──────────────────────────────────────╮
│ embedded    5 files    12 chunks     │
╰──────────────────────────────────────╯
```

**Compact, validation failure** — steps appear because there's an error:
```
Scanned 5 files
Validated 5 files — 1 violation(s)

╭──────────────────────────────────────╮
│ "tags"   MissingRequired   1 file    │
╰──────────────────────────────────────╯
```

**Compact, infra error** — you see exactly where it broke:
```
Scanned 5 files
Validated 5 files — no violations
Loading model — failed: could not find "minishlab/potion-base-8M"
```

**JSON** carries the same structure — steps with status, so a consumer knows exactly which phase failed and what the prior state was.

### Error rendering

Define a `CommandError` struct that renders consistently in both text and JSON:

```
// --output text
error: no markdown files found in 'path'

// --output json
{ "error": { "kind": "user", "message": "no markdown files found in 'path'" } }
```

Success output remains unchanged — no envelope wrapper.

### Categorization approach

- Define a `UserError` type for explicit `bail!()` sites (most current bail calls)
- Treat unrecognized `anyhow::Error` as application errors
- In `main.rs`, catch errors at the top level and render via `CommandError`

### Scope

- All 7 commands: init, check, build, search, update, info, clean
- Both `--output text` and `--output json`
- Exit codes: 0 for success, 1 for violations (normal result), 2+ for errors

## Files

- `src/output.rs` — `CommandError` struct, `ErrorKind` enum, rendering logic
- `src/main.rs` — top-level error catch and rendering
- `src/cmd/*.rs` — convert `bail!()` calls to `UserError` where appropriate
- `src/cmd/build.rs` — step-based output for build pipeline
- `src/cmd/init.rs` — step-based output for init pipeline
- `src/cmd/update.rs` — step-based output for update pipeline
