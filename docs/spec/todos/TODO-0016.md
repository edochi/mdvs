# TODO-0016: Replace Parquet + DataFusion with Lance + LanceDB

## Summary

Swap the storage and search engine entirely from Parquet + DataFusion to
Lance + LanceDB. **Not** a parallel backend; the Parquet codepath is
deleted in the same change. Includes native ANN indexing (IVF-PQ), BM25
full-text indexing, and **hybrid search (vector + BM25 + RRF) as the
default search mode**. New `--mode semantic|fulltext|hybrid` flag exposes
the three retrieval modes; default is `hybrid`.

This is the third and final launch-readiness item for the v0.6 / v1.0
release.

## Goals

- **Offline + embeddable** — LanceDB OSS mode runs in-process, persists to
  the local filesystem. No service, no daemon. Same posture as today.
- **No feature regressions** — every `--where` operator we support today
  keeps working: comparison ops, `IN`, `BETWEEN`, `IS NULL`, `LIKE`,
  `AND`/`OR`, `array_has(col, val)`, struct/dotted-leaf access, date and
  timestamp literals.
- **Performance** — ANN replaces brute-force cosine for vector search at
  scale; brute force remains automatic up to a few hundred thousand
  vectors. Build pipeline remains incremental (content-hash diff).
- **Hybrid search** — vector + BM25 + RRF reranking in a single query path,
  with model2vec embeddings supplied explicitly (bring-your-own-embedding
  pattern).

## Non-goals

- **Migration story for existing indexes.** mdvs has no users yet; old
  `.mdvs/files.parquet` directories will simply emit "rebuild required"
  on first run. No grace period, no auto-migration.
- **Compile-time feature flag for choosing between backends.** Earlier
  draft of this TODO proposed `--features lancedb` with `ParquetBackend`
  as the default. Decision (2026-05-21): full swap, single backend.
  Maintenance surface stays small, dependency footprint shrinks.
- **Custom rerankers** beyond RRF in v0.6. LanceDB ships RRF as the
  default reranker; that's enough. Custom rerankers are a separate TODO
  if anyone asks for them.

## Design decisions

### Path: denormalize into a single Lance table

LanceDB's query API is single-table-oriented (no JOIN primitive). Instead
of two tables joined at search time (the current `files.parquet` +
`chunks.parquet` shape), the index becomes **one Lance table** where
each row is one chunk **with the file's metadata duplicated inline**.

Storage cost is small in practice — Lance's columnar encoding
run-length-compresses the repeated `filepath` and frontmatter values for
chunks belonging to the same file. Embedding bytes dominate the on-disk
size regardless. Expected growth: 5–15% vs today's two-Parquet layout.

Query benefits:
- The JOIN disappears from the search SQL.
- The custom `cosine_similarity` UDF disappears — LanceDB's vector
  search is a built-in API call.
- "Best chunk per file" (today's `ROW_NUMBER() OVER (PARTITION BY ...)`)
  is post-processed in Rust by grouping the top-K result rows by
  `file_id`.

### Schema (single Lance table `index`)

| Column | Arrow type | Notes |
|---|---|---|
| `chunk_id` | `Utf8` | UUID, primary identity |
| `file_id` | `Utf8` | Stable across rebuilds for unchanged files |
| `chunk_index` | `Int32` | 0-based position within the file |
| `start_line` | `Int32` | 1-based |
| `end_line` | `Int32` | 1-based, inclusive |
| `chunk_text` | `Utf8` | Plain text of the chunk (needed for FTS index) |
| `embedding` | `FixedSizeList<Float32, dim>` | Vector |
| `filepath` | `Utf8` | Duplicated per chunk |
| `content_hash` | `Utf8` | Duplicated per chunk |
| `data` | `Struct{...}` | Nested frontmatter Struct, duplicated per chunk |
| `built_at` | `Timestamp(Microsecond, UTC)` | Duplicated per chunk |

`chunk_text` is **new** in this iteration — required for the BM25 FTS
index. Today's chunks.parquet doesn't persist chunk text (it's
regenerated on demand). Lance keeps it.

### Build metadata

Stored in the Lance table's table-level metadata (verified during
wave 1). Keys are the same as today's Parquet kv-metadata:
`mdvs.provider`, `mdvs.model`, `mdvs.revision`, `mdvs.chunk_size`,
`mdvs.glob`, `mdvs.built_at`, `mdvs.schema_hash`. Schema-hash mismatch
between current config and stored metadata still requires
`mdvs build --force`.

### Search modes

New global flag `--mode <MODE>` on `mdvs search`, accepting one of:

- `semantic` (vector-only) — `table.query().nearest_to(embedding).limit(k)`
- `fulltext` (BM25-only) — `table.query().full_text_search(query).limit(k)`
- `hybrid` (default) — vector + BM25 + RRF reranker

`hybrid` is the default. The CLI surface is otherwise unchanged: same
`--where`, same `--limit`, same `--output`. Output format (table or
JSON) is unchanged regardless of mode — RRF returns one ranked list.

### `--where` clause translation

LanceDB documents two friction points; both are absorbed inside our
SQL validator so the user-facing surface is identical to today.

1. **Dotted-leaf struct access.** Today users write
   `--where "calibration.baseline.wavelength > 800"`. LanceDB requires
   backticks for nested-struct access:
   `` `calibration`.`baseline`.`wavelength` > 800``. We add a
   plain-dot → backtick translator in `cmd/search.rs` next to the
   existing quote-balance check. Tests cover round-trips for every
   nested-access pattern in the book.
2. **Operators.** All operators and functions we use today are
   supported by LanceDB/DataFusion: `=`, `<>`, `<`, `>`, `<=`, `>=`,
   `AND`, `OR`, `IN`, `BETWEEN`, `IS NULL`, `LIKE`, `array_has`,
   `array_has_any`, `array_has_all`, `date 'YYYY-MM-DD'` literals,
   `timestamp 'YYYY-MM-DD HH:MM:SS'` literals, `EXTRACT(... FROM ...)`,
   `date_part(...)`. The book's `search-guide.md` examples are exercised
   verbatim as integration tests in wave 2.

### Crate / dependency changes

Removed from `crates/mdvs/Cargo.toml`:

- `datafusion`
- `parquet`
- `arrow-array` (kept if needed transitively, no direct dep)

Added:

- `lancedb`
- `lance` (if direct API access needed beyond `lancedb`; otherwise
  transitive)

`tokio` remains (LanceDB is async). `arrow` stays for in-memory
RecordBatch construction.

## Architecture

### `IndexBackend` trait

Keeps the trait abstraction (useful for testability) but with a single
implementation `LanceBackend`. The `Backend::parquet()` constructor and
the `ParquetBackend` struct are deleted in wave 1. `src/search.rs`
(cosine UDF + SQL generation) is deleted in wave 2.

Trait surface stays the same shape, with one signature change:

```rust
async fn search(
    &self,
    query_embedding: Vec<f32>,
    query_text: &str,            // NEW — required for FTS / hybrid modes
    mode: SearchMode,             // NEW — Semantic | Fulltext | Hybrid
    where_clause: Option<&str>,
    limit: usize,
    internal_prefix: &str,
    aliases: &HashMap<String, String>,
) -> anyhow::Result<Vec<SearchHit>>;
```

`SearchHit` gains the chunk text (`chunk_text: Option<String>`) since
it's now persisted in the table.

### Build pipeline

`build` flow (`cmd/build.rs`) is unchanged at the boundary: scan +
validate + chunk + embed → emit `Vec<FileRow>` and `Vec<ChunkRow>` →
hand to backend. The backend:

1. Builds a single denormalized Arrow `RecordBatch` (joins file and
   chunk rows in memory).
2. Calls `Table::create_or_overwrite` on the Lance connection.
3. Creates / refreshes the IVF-PQ vector index on `embedding`.
4. Creates / refreshes the BM25 FTS index on `chunk_text`.
5. Writes table-level metadata via Lance's metadata API.

Index creation is idempotent — Lance handles "index already exists"
gracefully. Re-build either replaces the table outright or appends with
the same overwrite semantics; final wave 1 design pins which.

### Search path

For `mode = hybrid` (default):

```rust
table.query()
    .nearest_to(embedding)?
    .full_text_search(FullTextSearchQuery::new(query_text))
    .where_clause(translated_user_where)
    .rerank(RRFReranker::default())
    .limit(limit * over_fetch_factor)
    .execute()
    .await
```

Result rows are grouped by `file_id` in Rust; the highest-scoring row
per file becomes the `SearchHit`. The `over_fetch_factor` (initially 3)
compensates for chunk-level dedupe — if we want N file-level hits and
files have ~K chunks each, we need roughly N×K chunk-level hits.
Wave 2 tuning task: pick a sane default and document the tradeoff.

Modes `semantic` and `fulltext` use the same query builder, omitting
the FTS or vector clause respectively.

## Wave plan

### Wave 1 — Storage and build

**Scope.** Replace the storage layer end-to-end. Search still runs the
old way during wave 1 only via a temporary in-Rust brute-force loop
over the Lance table's `embedding` column.

- Add `lancedb` dependency.
- Implement `LanceBackend` against the `IndexBackend` trait.
- Build the denormalized RecordBatch in `LanceBackend::write_index`.
- Persist build metadata in Lance table metadata.
- Implement `read_metadata`, `read_file_index`, `read_chunk_rows`,
  `embedding_dimension`, `stats`, `exists`, `clean`.
- Temporary `search()` implementation: read all chunks, brute-force
  cosine in Rust, apply user `--where` via a small DataFusion-less
  filter, return top-K. (Throwaway code; deleted in wave 2.)
- Delete `ParquetBackend` and `Backend::parquet()` constructor.
- Delete the `parquet` crate dep; keep `datafusion` and `arrow` for one
  wave (still used by the wave-1 temporary search filter).
- Tests: parity on example_kb — `init`, `build`, `check`, `info`,
  `clean` all green.

**Acceptance**: `cargo test -p mdvs` green;
`cargo run -- build example_kb` produces `.mdvs/index.lance/`;
`cargo run -- search 'experiment' example_kb` returns the same top-K as
the pre-swap baseline (within float tolerance).

### Wave 2 — Native search, ANN, hybrid, mode flag, where-translator

- Build BM25 FTS index on `chunk_text` in `LanceBackend::write_index`.
- Add `--mode <semantic|fulltext|hybrid>` flag on `mdvs search`.
  Default `hybrid`. Clap derive update.
- Result dedupe by `file_id` (best chunk per file) in Rust.
- Delete `src/search.rs` (cosine UDF + SQL builder).
- Delete `datafusion` direct dep; verify nothing breaks transitively.

**Acceptance**: `cargo test -p mdvs` green; manual smoke on

## Open questions to settle during implementation

These are not blockers but need answers as the work progresses; capture
the decisions in commit messages or follow-up TODOs.

- **`over_fetch_factor` for chunk-to-file dedupe.** Pick a sane default
  in wave 2. Probably 3–5; benchmark on example_kb and Refractions.
- **ANN index parameters.** IVF-PQ has `num_partitions` and
  `num_sub_vectors`. LanceDB picks defaults based on dataset size;
  verify those defaults work on KBs of 10–100k chunks.
- **FTS tokenizer.** LanceDB's default tokenizer is "simple" (whitespace
  + lowercase). Confirm it's reasonable for English markdown; revisit
  if Refractions search quality dips.
- **Lance table metadata API specifics.** Verify in wave 1 that
  `Table::set_metadata` (or equivalent) supports the seven `mdvs.*`
  keys we persist today, and that reading them back is straightforward.
- **Async surfacing.** LanceDB is async; our commands are already async
  via `tokio`. Verify no rough edges where we previously used sync
  Parquet reads.
