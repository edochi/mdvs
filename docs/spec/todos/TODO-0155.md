# TODO-0155: Reusable type definitions via $defs / $ref in mdvs.toml

## Summary

Add a `[definitions]` section to `mdvs.toml` for named, reusable type definitions. Fields reference definitions by name. The translator emits these as JSON Schema `$defs` + `$ref`, the standard mechanism every JSON Schema tool supports.

Motivation: after Wave C (TODO-0097) flattens top-level Objects, the only way to express **arrays of structured items** is to inline the struct in the field's `type`:

```toml
[[fields.field]]
name = "readings"
type = { array = { object = { time = "String", value = "Float" } } }
```

This works but it's:

- Verbose for repeated shapes (e.g. multiple fields holding the same kind of struct).
- An asymmetric notation — top-level Objects use dotted-name leaves, Array-of-Object uses inline structs. Two ways to write a struct.
- Not what users coming from JSON Schema expect — `$defs` is idiomatic.

After this TODO:

```toml
[definitions.SensorReading]
type = { object = { time = "String", value = "Float" } }

[[fields.field]]
name = "readings"
type = { array = "@SensorReading" }   # reference syntax TBD
```

One definition, reusable everywhere. The translator emits `$defs.SensorReading` + `items.$ref = "#/$defs/SensorReading"`.

## Why this exists as a separate TODO

Conceptually independent from object flattening (TODO-0097):

- TODO-0097 removes top-level Object as a `[[fields.field]]` type. After it lands, mdvs has no top-level Object support and Array(Object) is inline-only.
- This TODO adds a new mechanism (`$defs`) that's useful regardless of flattening — even pre-flattening it would have helped with repeated nested shapes.

In practice ship order: Wave C (object flattening) → this TODO. Reason: flattening clarifies what we need definitions for. Pre-flattening, top-level Objects could already inline their shape; the pressure for reuse came from arrays.

## Design questions to resolve

### Reference syntax in `mdvs.toml`

JSON Schema uses `{"$ref": "#/$defs/Name"}`. TOML-side spellings under consideration:

| Spelling | Pros | Cons |
|---|---|---|
| `"@SensorReading"` | Short, visually distinct | New magic sigil |
| `"#/$defs/SensorReading"` | Identical to JSON Schema | Verbose, looks alien in TOML |
| `{ ref = "SensorReading" }` | Pure TOML, no sigils | Inline-table noise |
| `"SensorReading"` (any unknown string = ref) | Maximally minimal | Ambiguous with future scalar type names |

Default proposal: `"@SensorReading"`. Decided in the planning phase, not at impl time.

### Where does `[definitions]` live in the file?

`[definitions.<Name>]` as a top-level section, sibling to `[fields]`. Each entry is a `TomlField`-shaped record minus `name`, `allowed`, `required`, `nullable` (those don't apply to definitions — definitions are pure type shapes).

```toml
[definitions.SensorReading]
type = { object = { time = "String", value = "Float" } }

[definitions.Author]
type = { object = { name = "String", email = "String" } }
```

### What can be referenced?

Only the `type` slot accepts `@Name`. Constraints, allowed, required, nullable, preprocess all stay on the field, not on the definition. (Could be revisited later if definitions need their own constraints.)

### What can a definition contain?

For v0:

- Scalar types: `"String"`, `"Integer"`, `"Float"`, `"Boolean"`
- Array of scalar or `@Name`: `{ array = "String" }`, `{ array = "@OtherDef" }`
- Object with literal inner shape: `{ object = { ... } }`
- **No recursion** — a definition can't reference itself or form a cycle (validated at config load).
- **No nested $ref-in-Object-children** initially — keeps the design simple. The Object shape is a value, not a graph.

### Inference question

Should `mdvs init` auto-detect structurally-duplicated shapes across fields and synthesize `[definitions]` entries? Probably **no** for v0:

- Naming definitions is a judgment call — autogenerated names (`Shape1`, `Shape2`) are noise.
- The user knows their domain; they can hand-extract definitions during cleanup.
- Auto-extraction can land as a follow-up if users complain.

`mdvs init` continues to emit inline structs for Array-of-Object fields. The user manually refactors into definitions when they see duplication.

### Gate updates

`schema/json_schema.rs::validate_mdvs_schema` currently rejects `$ref` and `$defs` in the hard-reject list. This TODO **lifts those rejections** but adds new constraints:

- `$ref` must point to `#/$defs/<Name>` only — no `#/properties/...`, no remote URIs, no recursive references.
- `$defs` must be a flat map of name → schema.
- Cycle detection: definition graph is a DAG. Validated at config-load via topological sort.

The translator's gate becomes a allow-list of `$ref`/`$defs` shapes rather than blanket rejection.

## Files touched

### Code

- **`crates/mdvs/src/schema/config.rs`** — new `definitions: BTreeMap<String, Definition>` field on `MdvsToml`. New `Definition` struct (subset of `TomlField`: just `type`, no field-level metadata). Sixth `MdvsToml::validate()` invariant: definition names well-formed, references resolve, no cycles.
- **`crates/mdvs/src/schema/json_schema.rs`** — `dsl_to_canonical` emits `$defs` and `$ref`. `canonical_to_dsl` reads them back. `validate_mdvs_schema` allow-list updated. Strict pattern matching for the reverse translator updates.
- **`crates/mdvs/src/discover/infer/`** — inference produces inline structs (no change). When/if auto-extraction lands, it goes here.
- **`crates/mdvs/src/cmd/init.rs`** — `--from-jsonschema` reads `$defs` and reproduces them in the `[definitions]` section.
- **`crates/mdvs/src/cmd/export_jsonschema.rs`** — already emits valid JSON Schema; just verifies definitions round-trip.
- **`crates/mdvs/src/cmd/check.rs`** — no change in the validation loop (`jsonschema::Validator` already resolves `$ref` internally). Per-field validator compilation handles the resolved schema.
- **`crates/mdvs/src/preprocess.rs`** — `strict_subtype_check` and the pipeline need to handle definitions: when a field's `type` is `@SensorReading`, the strict-Float / coerce-to-string checks recurse into the definition's resolved shape (since the gate prevents cycles, recursion terminates).

### Example_kb

After this lands, consider adding a field that demonstrates the pattern. Candidates:
- `readings` (Array(Object)) on experiment files — sensor time-series snapshots.
- `attendees` (Array(Object)) on meeting files — name + role.

These show the feature's value without bloating the example.

## Out of scope

- Auto-extraction of duplicated shapes during inference.
- Constraints on definitions themselves (e.g. a definition with its own `categories`). For v0, constraints stay on the field.
- Remote `$ref` (other files, URLs). Local `#/$defs/...` only.
- Recursive types (linked-list, tree). Cycle detection rejects them.

## Definition of done

- `[definitions]` section parses, validates, and round-trips through `dsl_to_canonical` ↔ `canonical_to_dsl`.
- `$ref` to `#/$defs/<Name>` is accepted by the gate; other `$ref` forms remain rejected.
- Strict subtype prechecks and preprocessor pipeline recurse into definitions when a field's type is a reference.
- example_kb can express its structured arrays via definitions if any are added (not required, but possible).
- Specs + mdBook describe the feature.
