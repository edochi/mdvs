# Configuration: `frontmatter.toml`

**Status: DRAFT**

**Cross-references:** [Terminology](../01-terminology.md) | [Crate: mdvs-schema](../10-crates/mdvs-schema/spec.md) | [Crate: mfv](../10-crates/mfv/spec.md)

---

## Overview

Field schema file shared between `mfv` and `mdvs`. Defines frontmatter field types, validation rules, and promotion flags. Generated by `mfv init` or `mdvs init`, then edited by the user to add validation rules.

**Location:** `frontmatter.toml` in the vault root (or specified via `--schema`).

---

## File Format

### Top-Level Sections

```toml
[directory]
glob = "**/*.md"

[fields.<name>]
# field definitions (one section per field)
```

### `[directory]` Section

| Key | Type | Default | Description |
|---|---|---|---|
| `glob` | string | `"**/*.md"` | File glob pattern for discovery and validation |

### `[fields.<name>]` Sections

Each field is defined under `[fields.<name>]` where `<name>` matches the frontmatter key.

| Key | Type | Default | Description |
|---|---|---|---|
| `type` | string | (inferred) | Explicit type override. One of: `string`, `string[]`, `date`, `boolean`, `integer`, `float`, `enum` |
| `required` | bool | `false` | Field must be present |
| `paths` | string[] | `[]` (all files) | Glob patterns where this field's rules apply |
| `pattern` | string | — | Regex the value must match (strings only) |
| `values` | string[] | — | Allowed values (only valid when `type = "enum"`) |
| `promoted` | bool | `false` | mdvs-specific: becomes a typed SQL column. Ignored by `mfv`. |

All keys are optional. A minimal field definition is just the section header:

```toml
[fields.title]
```

This declares that `title` exists. Its type will be inferred from observed values. No validation rules, not promoted.

## Type System

### Supported Types

| Type | YAML Example | DuckDB Mapping | Notes |
|---|---|---|---|
| `string` | `title: "My Post"` | `VARCHAR` | Default fallback |
| `string[]` | `tags: [rust, crdt]` | `VARCHAR[]` | YAML sequences |
| `date` | `date: 2025-06-12` | `DATE` | ISO 8601 dates |
| `boolean` | `draft: true` | `BOOLEAN` | YAML booleans |
| `integer` | `priority: 3` | `BIGINT` | YAML integers |
| `float` | `score: 0.95` | `DOUBLE` | YAML floats |
| `enum` | `status: draft` | `VARCHAR` | Validated against `values` list |

### Type Inference

When `type` is omitted, it's inferred from observed values during `init`. See [mdvs-schema: Type Inference](../10-crates/mdvs-schema/spec.md#type-inference) for the full rules.

The inferred type can always be overridden by setting `type` explicitly.

## Validation Rules

### `required`

```toml
[fields.title]
required = true
```

The field must be present in every matching file's frontmatter. If `paths` is set, the requirement only applies to files matching those patterns.

### `paths`

```toml
[fields.status]
required = true
paths = ["blog/**"]
```

Scopes when the field's rules apply. Glob patterns are relative to the vault root. If omitted or empty, rules apply to all files.

Multiple patterns are OR-ed: a file matching any pattern is in scope.

### `pattern`

```toml
[fields.doi]
type = "string"
pattern = "^10\\.\\d{4,9}/.*"
```

Regex the string value must match. Only valid for `string` type fields. Uses Rust regex syntax.

### `values`

```toml
[fields.status]
type = "enum"
values = ["draft", "review", "published"]
```

Allowed values for enum fields. Only valid when `type = "enum"`. The value must exactly match one of the listed strings.

### `promoted`

```toml
[fields.tags]
promoted = true
```

mdvs-specific flag. When `true`, this field becomes a typed column on the `mdfiles` DuckDB table instead of being stored in the JSON `metadata` column. Completely ignored by `mfv`.

## Examples

### Minimal (promoted fields only, no validation)

```toml
[directory]
glob = "**/*.md"

[fields.title]
promoted = true

[fields.tags]
promoted = true

[fields.date]
promoted = true
```

### Full-featured

```toml
[directory]
glob = "**/*.md"

[fields.title]
promoted = true

[fields.tags]
promoted = true

[fields.date]
promoted = true

[fields.status]
type = "enum"
values = ["draft", "review", "published"]
required = true
paths = ["blog/**"]

[fields.doi]
type = "string"
required = true
paths = ["papers/**"]
pattern = "^10\\.\\d{4,9}/.*"
promoted = true

[fields.category]
type = "string"
```

### Validation-only (for `mfv` users without `mdvs`)

```toml
[directory]
glob = "content/**/*.md"

[fields.title]
required = true

[fields.date]
type = "date"
required = true

[fields.draft]
type = "boolean"
```

No `promoted` flags needed since `mfv` ignores them.

## Generation Behavior

### `mfv init`

Generates `frontmatter.toml` with:
- `[directory]` section with the glob pattern
- One `[fields.<name>]` section per discovered field
- Inferred types (no explicit `type` unless inference is ambiguous)
- No validation rules (user adds them manually)
- No `promoted` flags

### `mdvs init`

Generates `frontmatter.toml` with:
- Same as `mfv init`, plus
- `promoted = true` on user-selected fields

### Existing File

If `frontmatter.toml` already exists when `mdvs init` runs, it reads the existing file and adds `promoted` flags for the selected fields. It does not re-scan or overwrite other settings.

If `frontmatter.toml` already exists when `mfv init` runs, it prompts for confirmation before overwriting (or errors without `--force`).
